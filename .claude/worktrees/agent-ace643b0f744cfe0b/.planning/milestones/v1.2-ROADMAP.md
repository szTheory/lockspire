# Milestone v1.2: PAR Foundation

**Status:** ✅ SHIPPED 2026-04-24
**Phases:** 14-16
**Total Plans:** 8

## Overview

Lockspire completed its third planning milestone by adding pushed authorization requests as a narrow extension of the existing authorization code + PKCE provider surface, then closing the remaining release-runtime hygiene gap that the prior milestone intentionally deferred. The result is truthful PAR support limited to Lockspire-issued `request_uri` references, with discovery, docs, tests, and release guidance aligned to the shipped slice.

This archive records the milestone as complete from a planning and implementation standpoint. Package version `0.2.0` remains preview posture at archive time, but the shipped surface now includes the PAR wedge and a checked-in Release Please path that no longer depends on the deprecated Node 20 marketplace action runtime.

## Milestone Summary

**Key Accomplishments:**

- Added durable, hash-at-rest PAR request intake with opaque Lockspire-issued `request_uri` references and a mounted `POST /par` endpoint.
- Extended `/authorize` to consume PAR references with expiry checks, wrong-client burn semantics, single-use enforcement, and unchanged auth-code + PKCE flow semantics.
- Published discovery metadata and support docs that describe only the shipped PAR slice and explicitly avoid broader request-object, DCR, device-flow, or hosted-auth claims.
- Added browser, integration, discovery, and repo-truth contract coverage that proves the `/par -> /authorize -> /token` path and rejects replay or stale/wrong-client references safely.
- Closed milestone traceability with explicit `PAR-04` verification artifacts instead of duplicating an already-sufficient proof stack.
- Removed the known Release Please runtime deprecation risk by moving the workflow to a checked-in Node 24 composite action without widening the protected release contract.

**Key Decisions:**

- Keep v1.2 narrow around Lockspire-issued PAR rather than broadening into JAR-by-value, DCR, device flow, or sender-constrained token work.
- Store only hashes of PAR `request_uri` values at rest so durable state never keeps the opaque reference in plaintext.
- Burn PAR references transactionally inside storage semantics, including wrong-client and expired paths, so replay resistance does not depend on controller behavior.
- Keep discovery truth route-derived and describe PAR publicly only as Lockspire-issued `request_uri` support on the existing authorization code + PKCE path.
- Wrap Release Please in a repo-owned composite action so future runtime migrations stay behind the same checked-in workflow contract.

**Issues Resolved:**

- Added the missing PAR capability that v1.1 intentionally deferred to the next milestone.
- Closed truthful-support drift by pinning PAR claims in discovery, docs, security wording, and release-readiness contract tests.
- Removed the deferred Node.js 20 Release Please runtime warning from the checked-in preview release lane.

**Issues Deferred:**

- No additional deferred milestone gaps remain in the formal audit; v1.2 passed with full requirements, integration, flow, and Nyquist coverage.
- Future protocol-expansion work such as broader JAR support, PAR policy controls, DCR, device flow, and sender-constrained token modes remains for later milestone planning.

**Technical Debt Incurred:**

- Milestone close still depends on the manual archive flow because `gsd-sdk query milestone.complete` remains unreliable in this repository workflow.
