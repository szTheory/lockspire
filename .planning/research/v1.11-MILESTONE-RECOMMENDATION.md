# Lockspire v1.11 Milestone Recommendation

## Executive Summary

After a deep review of Lockspire's current trajectory (via `EPIC.md` and `PROJECT.md`), the most aggressive and cohesive way to move Lockspire toward true production readiness for real Phoenix SaaS apps is **not** to add more protocol breadth, but to declare the preview phase over. 

The recommended v1.11 milestone is: **1.0 General Availability & Production Operator Tooling**.

Lockspire has already shipped a staggering amount of high-value protocol surface (PAR, DCR, Device Flow, DPoP, JWE, FAPI 2.0). The biggest blocker to widespread adoption right now is the perceived risk of a `0.x` "preview" auth library. By focusing entirely on API stabilization, operational observability, and best-in-class operator DX, we transition Lockspire from a "promising preview" into a "boring, trustworthy production asset."

---

## 1. Analysis of Candidate Paths

### Candidate A: mTLS-Bound Tokens (RFC 8705)
**The Concept:** Binding tokens to client X.509 certificates to prevent token theft (Sender-Constraining).
* **Pros:** The gold standard for financial/high-security APIs. Required for FAPI 1.0 Advanced.
* **Cons & Footguns:** Massive operational complexity. Standard Phoenix apps are deployed behind reverse proxies (Nginx, AWS ALB, Cloudflare, Fly.io). Terminating mTLS requires proxy-level configuration to forward the certificate or its hash via headers (e.g., `Client-Cert` headers) down to the Elixir app. 
* **Elixir/Phoenix Idioms:** While Cowboy *can* terminate mTLS, doing so in a containerized PaaS environment is non-standard. It violates the "embedded library" thesis by forcing the host team to rewrite their infrastructure layer.
* **Verdict:** **DEFER.** Lockspire already shipped DPoP (application-layer sender constraining) which solves token theft with far better developer ergonomics for standard SaaS apps.

### Candidate B: Rich Authorization Requests (RAR - RFC 9328)
**The Concept:** Replacing static scopes (`read`, `write`) with fine-grained, dynamic JSON payloads (e.g., "transfer $100 from Account A to B").
* **Pros:** Highly modern; solves the "scope explosion" problem. Pairs beautifully with Lockspire's existing PAR implementation.
* **Cons & Footguns:** Consent UX becomes highly complex. The host app must dynamically render consent screens based on arbitrary JSON. 
* **Elixir/Phoenix Idioms:** Ecto's `{:map, ...}` and Postgres `jsonb` handle the data model effortlessly, and Phoenix LiveView is perfectly suited for dynamic consent screens.
* **Verdict:** **DEFER UNTIL POST-1.0.** RAR is an excellent feature for fintech/healthtech, but it is not a blocker for 95% of standard SaaS identity providers. 

### Candidate C: 1.0 General Availability & Hardening (The Winner)
**The Concept:** Freezing the API contract, polishing the operator tooling, and cementing the release as production-ready.
* **Pros:** Directly attacks the `EPIC.md` mandate: *"The priority order is: 1. Raise real-client trust on the surfaces already shipped."* 
* **Cons & Footguns:** Lacks a "shiny new protocol spec" to tweet about, requiring deep, unglamorous engineering discipline (load testing, telemetry, audit logs, API stability).
* **Verdict:** **DO THIS NOW.** This is the exact right unit of work to unlock downstream adoption.

## 2. Lessons Learned from the Ecosystem

If Lockspire is going to be the standard Elixir OIDC provider, we must learn from the giants:

1. **node-oidc-provider (Node.js)**
   * *What they did right:* Uncompromising protocol strictness. If a client deviates from the spec, it fails securely. 
   * *The Footgun:* Configuration hell. Passing massive nested JavaScript objects with dozens of callbacks is terrible DX. 
   * *Lockspire Takeaway:* Keep configuration declarative and backed by Ecto. Rely on explicit Plug pipelines rather than massive config maps.
2. **OpenIddict (C# / .NET)**
   * *What they did right:* Impeccable separation of concerns. The core protocol has zero knowledge of the web framework, and the storage layer is heavily abstracted.
   * *The Footgun:* Can feel overly enterprisey and boilerplate-heavy to get started.
   * *Lockspire Takeaway:* Our generators (`mix lockspire.install`) must continue to output clean, readable Phoenix code that the host app owns.
3. **Doorkeeper (Ruby on Rails)**
   * *What they did right:* Unbelievably good "Time to First Token" (TTFT). You run a generator, run migrations, and you have an OAuth server.
   * *The Footgun:* Hard to extend securely when you outgrow the defaults; heavily coupled to Rails conventions.
   * *Lockspire Takeaway:* Keep the Doorkeeper-style onboarding DX, but back it with node-oidc-provider's protocol strictness.

## 3. The Recommended Milestone: 1.0 GA & Production Operator Tooling

To aggressively move Lockspire into production readiness, **Milestone v1.11** should focus on transforming Lockspire into a robust, observable, and easily administrable system.

### The "Meaty" Scope of Work

**1. API Contract & Database Schema Lockdown**
* **Action:** Audit all public-facing Elixir APIs (`Lockspire.Clients`, `Lockspire.Tokens`, etc.) and finalize function signatures.
* **Action:** Finalize all Ecto migrations. Ensure indices are optimized for high-volume token lookups and purges.
* **Why:** 1.0 means no breaking changes. Integrators need this guarantee to adopt.

**2. Production-Grade Operator LiveViews (The "Auth0-like" Admin Experience)**
* **Action:** Elevate the generated Admin UI from "basic CRUD" to a production support tool.
* **Features:**
  * Rich search/filtering for Clients and Users (e.g., "Find all tokens for User X").
  * 1-click Revocation cascades (revoke an authorization code, which revokes the refresh token and access tokens).
  * Real-time LiveView audit logs of failed authentications, DPoP validation failures, and PAR rejections.
* **Why:** When production auth breaks, the SaaS support team needs tools, not a database console.

**3. "Agent-First" Observability & Telemetry**
* **Action:** Instrument the entire protocol core with `:telemetry` events (e.g., `[:lockspire, :token, :issued]`, `[:lockspire, :dpop, :failed]`).
* **Action:** Provide a generated, out-of-the-box Grafana/LiveDashboard module so operators can see token issuance rates, error rates, and key rotation metrics instantly.
* **Why:** Protocol correctness is invisible without telemetry. Principle of least surprise dictates that an auth library should tell you when it's under attack.

**4. Performance, Load Testing & Automated Pruning**
* **Action:** Implement and document an `Oban` worker (or standard GenServer) for aggressively pruning expired tokens, nonces, and orphaned PAR requests.
* **Action:** Benchmark the FAPI 2.0/DPoP pathways to ensure cryptographic operations (like JWE decryption) don't bottleneck the Phoenix request pool.

### Why this unlocks downstream work
By shipping 1.0 GA with world-class operator tooling, you remove the trust barrier for Phoenix developers. Once real applications are using Lockspire in production, the demand for post-1.0 features (like RAR, CIAM federation, or mTLS) will be driven by *actual user feedback* rather than speculative implementation. This cements Lockspire as the premier, trustworthy identity library for the Elixir ecosystem.
