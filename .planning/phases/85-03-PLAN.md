<objective>
Make DCR success and read responses expose the stored logout propagation metadata truthfully so self-service clients see the same persisted values that Lockspire will later use at runtime.
</objective>

<scope>
This plan covers representation and readback truth for DCR JSON surfaces. RFC 7592 full-replace update semantics remain deferred to Phase 86.
</scope>

<threat_model>
## Threats

- `T-85-05` DCR responses can drift from persisted client state, causing self-service clients to misread their registration contract.
- `T-85-06` Representation work could accidentally imply unsupported RFC 7592 write semantics before Phase 86 lands.

## Mitigations

- Serialize the four fields directly from the stored `Client`.
- Add read-path proof for operator-seeded and self-registered clients alike.
- Keep management update behavior out of scope for this phase; only read/response truth changes.
</threat_model>

<tasks>
- [ ] `85-03-01` Extend `Lockspire.Web.RegistrationJSON` to emit the four logout propagation fields from the persisted client when present.
  - Files: `lib/lockspire/web/registration_json.ex`, `test/lockspire/web/registration_json_test.exs`
  - Action: include the four fields in create/read/update payload shaping as stored client state, preserving omission-vs-present semantics where appropriate.
  - Verify: serializer tests assert both presence and absence cases.

- [ ] `85-03-02` Add end-to-end create/read proof for DCR logout metadata representation.
  - Files: `test/lockspire/web/controllers/registration_controller_test.exs`, `test/lockspire/protocol/registration_management_test.exs`
  - Action: prove a successful registration response includes accepted logout metadata and a subsequent RFC 7592 read returns the same persisted values.
  - Verify: targeted controller and management tests pass.

- [ ] `85-03-03` Prove read truth for already-persisted clients, not just DCR-created ones.
  - Files: `test/lockspire/protocol/registration_management_test.exs`
  - Action: add a case where logout propagation fields are already present on the stored client and confirm `RegistrationManagement.read/2` plus JSON serialization reflects them unchanged.
  - Verify: `DCRM-01` is satisfied without depending on Phase 86 update support.
</tasks>
