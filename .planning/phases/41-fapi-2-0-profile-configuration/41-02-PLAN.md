<objective>
Create the missing `Lockspire.Protocol.FAPI20EnforcerPlug` — the boundary fail-fast
enforcer locked in CONTEXT.md D-02 — and wire it into the Phoenix router. The Plug
performs the cheapest possible check at the earliest possible moment for the three
FAPI 2.0 boundary requirements: PAR presence on `/authorize`, DPoP header presence on
`/token`, DPoP header presence on `/userinfo`. The Plug is exempt for `/par` (the PAR
endpoint itself, by definition has no `request_uri` yet) and bypasses non-FAPI paths.

Purpose: FAPI-02 and FAPI-03 demand boundary enforcement so non-compliant requests are
rejected before any protocol work runs. The existing protocol modules already perform
defence-in-depth checks via `SecurityProfile.Resolved.fapi_2_0_security?`, but the
boundary Plug is the canonical fail-fast gate per the locked architecture decision in
CONTEXT.md ("A dedicated Plug acts as a centralized, fail-fast gateway, keeping base
OIDC modules clean while aggressively rejecting non-compliant requests"). This plan
delivers that Plug.

<userinfo_strategy>
<!-- Per RESEARCH.md Pitfall 2, /userinfo has no client_id in params. Decision: -->

The Enforcer enforces /userinfo DPoP via the **header-shape rule**, not the
client-resolution rule. Specifically: when ServerPolicy.security_profile is
`:fapi_2_0_security` (the global), the Enforcer rejects ANY /userinfo request lacking a
DPoP header or using the Bearer scheme. This is correct because:

  1. Under a global `:fapi_2_0_security`, ALL clients (regardless of override) issue DPoP-bound tokens
     unless explicitly overridden to `:none`. A bearer access-token presented at /userinfo under
     global `:fapi_2_0_security` is by definition non-conformant.
  2. The per-client override case (client has `security_profile: :fapi_2_0_security` overriding
     global `:none`) is handled by the existing `Lockspire.Protocol.Userinfo.fetch_claims/1`
     defense-in-depth path (see `validate_access_mode/4` which inspects `access_token.cnf["jkt"]`
     AND the resolved per-client profile). The Plug is the FAST PATH; the defense-in-depth is the
     CORRECT path for this layering.
  3. This avoids decoding/validating the access token inside the Plug just to identify the client
     — a layering violation called out in RESEARCH.md.

Net effect: under global `:fapi_2_0_security`, /userinfo requires a DPoP header at the boundary;
under global `:none`, the Enforcer is a no-op and per-client `:fapi_2_0_security` overrides are
caught by `Userinfo.fetch_claims/1`'s existing logic. This is acceptable per CONTEXT.md
("aggressive rejection") because the strict global is the recommended FAPI 2.0 deployment.

**Verification of the per-client opt-in path for /userinfo (defense-in-depth):** The Plug is
deliberately a no-op for this case (global `:none`); the rejection happens in
`Userinfo.fetch_claims/1`. That defense-in-depth path is verified by an integration test in
**Plan 04 Task 1** ("/userinfo defense-in-depth: global :none + per-client :fapi_2_0_security
+ Bearer access token + no DPoP header is rejected"). Plan 02's unit tests do NOT cover this
case (correctly — the Plug should not act on it); only Plan 04's integration test exercises
the cross-tier defense-in-depth path. See Plan 04 Task 1 acceptance criteria.
</userinfo_strategy>

<documented_limitations>
<!-- Plug responsibility boundaries that are intentionally narrow. -->

**Limitation 1: /token per-client opt-in via Authorization: Basic auth is NOT enforced by the Plug.**

The Plug only extracts `client_id` from `conn.params` (i.e. the request body for `POST /token`).
A non-FAPI client that authenticates at `/token` using `Authorization: Basic <base64(client_id:secret)>`
and does NOT include `client_id` in the body will silently bypass the per-client FAPI override at
the Plug layer. Concretely: under global `:none`, a client with override `:fapi_2_0_security` that
uses Basic auth at `/token` will pass through the Plug without DPoP enforcement.

**Why this is acceptable:** Defense-in-depth at `Lockspire.Protocol.TokenExchange` handles the
Basic-auth case via the existing per-client `security_profile` check on the resolved profile
(TokenExchange decodes the Authorization header, resolves the client, and checks DPoP binding
itself when the resolved profile is `:fapi_2_0_security`). The Plug is the FAST PATH for the
common case (client_id in body); TokenExchange is the CORRECT PATH for all cases including
Basic auth.

**Why we keep the Plug responsibility narrow:** Extending the Plug to decode `Authorization: Basic`
duplicates the credential-extraction logic that TokenExchange already implements correctly. That
duplication would create a security-relevant "two implementations of the same parser" hazard —
explicitly forbidden by CONTEXT.md D-02 ("Plug is the centralized, fail-fast gateway, keeping
base OIDC modules clean"). The Plug stays narrow; TokenExchange is the source of truth for
credential resolution at /token.

**Verification:** This defense-in-depth path is exercised by an integration test in
**Plan 04 Task 1** ("/token defense-in-depth: global :none + per-client :fapi_2_0_security +
Authorization: Basic + no DPoP header is rejected by TokenExchange (not the Plug)"). The Plug's
unit tests in this plan do NOT cover Basic auth on /token (correctly — the Plug is a no-op for
that path); Plan 04 Task 1 is the load-bearing verification for this limitation.
</documented_limitations>
