<objective>
Verify and complete the existing uncommitted FAPI 2.0 end-to-end integration test
(`test/integration/phase41_fapi_2_0_e2e_test.exs`) so it exercises the new Plug
boundary alongside the in-protocol defense-in-depth, replace the placeholder
conformance shell script with a script that runs actual `curl` probes against a
live server, and update `docs/maintainer-conformance.md` to describe the full
conformance workflow.

Purpose: The phase goal is a verifiable promise that "requests without PAR, requests
missing DPoP/mTLS, and non-S256 PKCE are aggressively rejected at the Plug boundary."
That promise needs (a) an automated integration test that hits the real router and
asserts the rejections at BOTH the boundary Plug AND the in-protocol defense-in-depth
tiers, and (b) a script a maintainer can run before declaring a FAPI 2.0 release. This
plan delivers both, and updates the docs so the workflow is discoverable. It also adds
two override-mode integration tests that close the remaining T-41-03 / T-41-07 evidence
gap, and three cross-tier defense-in-depth integration tests (T-41-16, T-41-17 from
Plan 02's narrow Plug responsibility).

  per-client override + mixed-mode bypass test cases + cross-tier defense-in-depth tests).
- Functional conformance shell script with executable bit set, real curl probes,

  registered as a Manual-Only Verification in 41-VALIDATION.md.

<conformance_script_design>
The current `scripts/conformance/fapi2-check.sh` is purely advisory text. Replace with
a real probe script that:
1. Reads `LOCKSPIRE_BASE_URL` env var (default `http://127.0.0.1:4000/lockspire`).
2. Reads `LOCKSPIRE_CLIENT_ID` env var (caller-provided).
3. Sends three probes via `curl -s -o /tmp/lockspire-fapi-out -w "%{http_code}"`:
   - GET `${BASE}/authorize?client_id=${CID}&response_type=code&redirect_uri=https://example.com/cb&scope=openid&code_challenge=abc&code_challenge_method=S256`
     — expect 302 with Location containing `error=invalid_request`.
   - POST `${BASE}/token` with form `grant_type=authorization_code&...` and NO `dpop` header
     — expect 400 with body containing `invalid_dpop_proof` or `invalid_request`.
   - GET `${BASE}/userinfo` with `Authorization: Bearer fake` — expect 401.
4. Report PASS/FAIL per probe with green/red ANSI colour.
5. Exit 0 only when all three probes pass.
6. Print a final pointer to `docs/maintainer-conformance.md` for the OIDF container workflow.

The script is best-effort observable — it requires the maintainer to first set the
ServerPolicy to `:fapi_2_0_security` via `Admin.put_security_profile/1` (either via the
LiveView at /admin/policies/security-profile or via `iex` REPL). The full prerequisites
and verification steps are documented in 41-VALIDATION.md "Manual-Only Verifications"
and in `docs/maintainer-conformance.md`.
</conformance_script_design>
