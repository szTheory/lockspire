# Phase 99 — UI Design Contract

> Visual and interaction contract for the Phase 99 frontend surface. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope note (read first):** This is a backend-heavy phase with a small, additive frontend surface inside the EXISTING admin client-detail LiveView. There is no greenfield design. The new control MUST mirror the established `dpop_policy` / `par_policy` / `security_profile` override-select idiom field-for-field. Do NOT introduce new visual language, new spacing scale, new typography, new colors, or new component patterns. The entire UI surface is: (1) one per-client `access_token_format` override `<select>` on the FORM, (2) a doclink beside it, and (3) global / override / effective display rows on the SHOW page. Everything inherits the existing admin stylesheet.

## Interaction Contract (the load-bearing part of this spec)

### Surface 1 — the override `<select>` (FORM)

Mirror the `dpop_policy` control at `form_component.ex:95-106` exactly, including the `:if={@mode == :edit}` gating, the bare `<label>` + `<select>` structure, and the `selected={@defaults.<field> == "<value>"}` option pattern.

**Field name (form param):** `client[access_token_format]` (mirrors `name="client[dpop_policy]"`).

**Options (exactly three, in this order):**

| Option `value` | Visible label (Claude's discretion per CONTEXT D-07) | Casts to |
|----------------|------------------------------------------------------|----------|
| `inherit` | `Inherit from server default` | `nil` (the inherit→nil mapping, D-06/D-07) |
| `jwt` | `JWT (RFC 9068 at+jwt)` | `:jwt` |
| `opaque` | `Opaque (Lockspire-stored)` | `:opaque` |

**`<label>` text (Claude's discretion):** `Access token format override` (parallels `Client DPoP override`).

> The option labels and field-label copy above are exercised under CONTEXT.md D-07's discretion grant. The locked constraints are: the three option `value`s (`inherit` / `jwt` / `opaque`) and the `inherit`→`nil` cast — those MUST NOT change.

**`defaults_for(:edit, client)` addition (`form_component.ex:406-416`):**
The `:edit` defaults map gains an `access_token_format` key. Because `nil` means inherit, the default-string conversion MUST map `nil → "inherit"` so the right option pre-selects. Pattern: `access_token_format: format_default_for_select(client.access_token_format)` where `nil → "inherit"`, `:jwt → "jwt"`, `:opaque → "opaque"`. (This differs from `dpop_policy`, which stores a real `:inherit` atom and can use a plain `Atom.to_string/1` — `access_token_format` stores `nil`, so a nil-aware helper is required. Closest behavioral precedent for the nil branch: `authorization_signed_response_alg` at `form_component.ex:427-429`, which guards nil before `Atom.to_string/1`.)

**Placement:** Inside the existing `<div :if={@mode in [:new, :edit]}>` block, immediately after the `dpop_policy` `<select>` (`form_component.ex:106`), so it reads as a sibling override control. Edit-mode only (`:if={@mode == :edit}`), matching `dpop_policy`.

### Surface 1b — the `normalize_mutable_field/2` clause (`admin/clients.ex`)

Add `access_token_format` to `@mutable_fields` (`clients.ex:14-33`) and add a `normalize_mutable_field(:access_token_format, value)` clause (`clients.ex:472-484` region). The normalizer MUST map the `"inherit"` string (and `nil`) to `nil`, and `"jwt"`/`"opaque"` to the corresponding atoms.

> **Precedent note for the executor:** This is a HYBRID of two existing normalizers. The SELECT/label idiom comes from `normalize_dpop_policy` (`clients.ex:575-590`, the `inherit` option), but the `inherit`→`nil` cast behaves like `normalize_authorization_signing_alg` (`clients.ex:611-629`, which maps `""`/`nil → {:ok, nil}`). Concretely: `"inherit" → {:ok, nil}`, `"jwt" → {:ok, :jwt}`, `"opaque" → {:ok, :opaque}`, `nil → {:ok, nil}`, anything else → `:error`. Do NOT store a `:inherit` sentinel atom — `nil` is the inherit state (D-06: nullable, no DB default).

### Surface 2 — the tradeoff doclink

A single inline link rendered beside the `<select>`, inside a `lockspire-admin-help` block (mirroring how `dpop_policy`'s neighbors and the security-profile help text use `class="lockspire-admin-help"`).

- **Target:** `docs/protect-phoenix-api-routes.md` (the Phase 97 D-06 canonical contract page). Locked by CONTEXT D-07.
- **Markup idiom:** plain `<a href={...}>` (the established admin doclink idiom — see `iat_live/new.html.heex:32` and the admin index views). Do NOT invent a new link component.
- **Anchor text (Claude's discretion):** `Learn when to choose JWT vs opaque` — concise, action-leading, names the tradeoff. See Copywriting Contract below for the surrounding help-text sentence.

### Surface 3 — global / override / effective display rows (SHOW)

Add three `<p>` rows to the SHOW template alongside the existing trios, mirroring the security-profile rows at `show.ex:169-171` and the PAR rows at `show.ex:204-206`:

| Row | Pattern (mirror) | Value shown |
|-----|------------------|-------------|
| Global access token format | `show.ex:169` (global security profile) | server-wide `ServerPolicy.access_token_format` (defaults `:jwt`) shown in `<code>` |
| Client access token override | `show.ex:170` (client security override) | per-client `@client.access_token_format` shown in `<code>`; render `nil` as a human label (recommend `inherit` to match how `par_policy_label`/`security_profile_label` stringify) |
| Effective access token format | `show.ex:171` (effective profile) | resolved effective format (per-client → server default → `:jwt`) shown in `<strong>` |

> **Nil-rendering note:** unlike `par_policy` / `security_profile` (which store a real `:inherit` atom and stringify cleanly), `access_token_format` stores `nil` for inherit. The override-row label helper MUST translate `nil → "inherit"` for display, paralleling the `value_or_not_configured/1` nil-guard idiom already in `show.ex:660-661` — but with the word `inherit`, not `Not configured`, since `nil` here means "inherit", not "unset".
