<objective>
Route the AC/device/CIBA mint seam through `AccessTokenSigner` (re-pointing the persisted `%Token{}.token_hash` to the signer's hash, NOT a literal one-liner) and add net-new `resource` validation + audience threading to the device and CIBA grant paths.

Purpose: SIGNER-01 (AC/device/CIBA call sites now call the shared signer), AUD-01 (device and CIBA `resource=` → `aud=[resource]` — genuinely new wiring, not propagation), AUD-02 (absent `resource=` → `aud=[client_id]`). AC already threads `resource` into `%Token{}.audience`, so AC's AUD-01/02 is satisfied once it mints through the signer; device and CIBA hardcode `audience: []` today and need a fresh `validate_requested_resources`-style guard.

Output: edits confined to `lib/lockspire/protocol/token_exchange.ex`. This plan owns that file exclusively in Wave 3; Plan 05 owns refresh + rfc8693 in parallel.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → token endpoint (device/CIBA) | The `resource` parameter is attacker-influenced and must be validated against the grant's authorized audience before it becomes the token's `aud`. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-99-11 | Elevation of Privilege | Audience confusion on device/CIBA — attacker requests `resource=` for an unauthorized API | mitigate | New `validate_requested_resources`-style guard rejects unauthorized resources with `invalid_target` (`:invalid_resource`, 400); only validated values reach `%Token{audience}`. The signer cannot be coerced into an attacker-chosen `aud` because it reads only the validated audience. |
| T-99-12 | Tampering / DoS on revocation | Stored hash ≠ hash-of-issued-token (Pitfall 1) | mitigate | `build_access_token/6` re-points `%Token{}.token_hash` to the signer's returned hash so introspection/revocation by hash resolves. A test asserts `token_hash == Policy.hash_token(issued_raw)`. |
| T-99-13 | Spoofing | Device/CIBA `resource=` silently ignored (left at `[client_id]`) | mitigate | Tests explicitly exercise a `resource=`-scoped device flow AND a `resource=`-scoped CIBA flow (Pitfall 2) — coverage proves the threading is live, not stubbed. |
| T-99-SC | Tampering | npm/pip/cargo installs | accept | No packages added in Phase 99. |
</threat_model>
