<objective>
Build a clean Elixir facade over Erlang's `:public_key.pkix_decode_cert/2` for MTLS certificate parsing.

Purpose: Erlang OTP provides robust X.509 parsing, but interacting with Erlang records in protocol logic is poor DX. This module shields the rest of the application from ASN.1 nuances.
Output: A parser that takes DER binary and returns an easy-to-use Elixir struct.
</objective>

<task type="tdd">
  <name>Task 1: MTLS Certificate Parser</name>
  <files>lib/lockspire/mtls/certificate.ex, test/lockspire/mtls/certificate_test.exs</files>
  <behavior>
    - Parses valid DER-encoded X.509 certificates returning `{:ok, struct}`.
    - Extracts `subject_dn` as a string (e.g., RFC 2253 formatted, or equivalent).
    - Extracts Subject Alternative Names (SANs) filtering by DNS, URI, IP, and Email into a nested `sans` map: `%{dns: [...], uri: [...], ip: [...], email: [...]}`.
    - Extracts the `public_key` in a standard format (e.g., the DER encoded public key or an Erlang key record suitable for conversion to JWK).
    - Handles parsing errors returning `{:error, reason}`.
  </behavior>
  <action>
    Follow the Facade Structure Pattern from `lib/lockspire/protocol/jwks.ex`.
    Use `Record` to extract `:OTPCertificate` and `:OTPTBSCertificate` from `"public_key/include/public_key.hrl". `:OTPCertificate` and `:OTPTBSCertificate` from `"public_key/include/public_key.hrl"`.
    In `parse/1`, call `:public_key.pkix_decode_cert(der, :otp)`. 
    Extract the subject DN.
    Look for the SAN extension with OID `{2, 5, 29, 17}`. If found, map the values to `dns` (dNSName), `uri` (uniformResourceIdentifier), `ip` (iPAddress), `email` (rfc822Name).
    Ensure the public key is extracted from `SubjectPublicKeyInfo`.
  </action>
  <verify>
    <automated>mix test test/lockspire/mtls/certificate_test.exs</automated>
  </verify>
  <done>The parser accurately extracts Subject DN and 4 relevant SAN types. The rest of the application interacts only with Elixir maps/structs, not Erlang records.</done>
