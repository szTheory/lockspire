<task type="tdd">
  <name>Task 1: MTLS Auth Verifier</name>
  <files>lib/lockspire/protocol/client_auth/mtls.ex, test/lockspire/protocol/client_auth/mtls_test.exs</files>
  <behavior>
    - `verify(client, mtls_cert_der, :tls_client_auth, opts)`:
      - Returns `:ok` if the certificate parses correctly and matches one of the client's registered PKI attributes (Subject DN, DNS, URI, IP, or Email).
      - Returns `{:error, reason}` if the cert is missing, malformed, or doesn't match the registered attributes.
    - `verify(client, mtls_cert_der, :self_signed_tls_client_auth, opts)`:
      - Converts the extracted cert public key to a JWK and checks if it exists in the client's JWKS.
      - Returns `:ok` if a match is found.
    - Emits telemetry for success/failure using Lockspire's standard auth telemetry formats.
  </behavior>
  <action>
    Follow the Auth Verifier Pattern from `PrivateKeyJwt`.
    First, check if `mtls_cert_der` is provided. If `nil`, immediately return `{:error, :missing_certificate}` (fail-closed).
    Parse the certificate using `Lockspire.MTLS.Certificate.parse/1`.
    For `:tls_client_auth`, compare the registered attribute on the client struct against the corresponding parsed SAN or Subject DN.
    For `:self_signed_tls_client_auth`, use `JOSE.JWK.from_key/1` to convert the parsed public key, and match it against the client's resolved JWKS.
    Record failures via `Observability.emit/4` using `[:client_auth, :mtls, :failure]`, etc.
  </action>
  <verify>
    <automated>mix test test/lockspire/protocol/client_auth/mtls_test.exs</automated>
  </verify>
  <done>Both MTLS methods validate correctly against mocked certificates. Missing certificates strictly fail. Telemetry events are fired upon resolution.</done>
