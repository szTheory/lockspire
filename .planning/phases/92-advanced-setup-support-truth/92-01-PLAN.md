Reconcile the shipped mTLS setup story across host-facing and maintainer-facing docs so adopters understand the exact extraction patterns, infrastructure prerequisites, and ownership split without inferring broader library guarantees.

Output: tightened mTLS setup docs and maintainer guidance that defer to the canonical support contract, with release-contract proof deferred to the canonical alignment plan.

<threat_model>
## Threats

- `T-92-01` mTLS docs could read like a library toggle and bury the host or proxy prerequisites needed to extract a trustworthy certificate.
- `T-92-02` Maintainer docs could restate or broaden mTLS support claims instead of deferring to the canonical contract.
- `T-92-03` Later proof work could miss the two-pattern extraction story or explicit ownership split if the doc changes are not handed off cleanly.

## Mitigations

- Teach the two shipped extraction patterns and their prerequisites up front in `docs/mtls-host-guide.md`.
- Repeat the proxy-header anti-spoofing warning where operators choose the proxy-header extractor.
- Keep maintainer docs deferential to `docs/supported-surface.md` and hand off the exact wording that Plan `92-03` must pin in release-contract tests.
</threat_model>
