<threat_model>
## Threats

- `T-92-04` Protected-route docs could continue implying `EnforceSenderConstraints` is merely optional, leading adopters to omit the canonical sender-constraint plug.
- `T-92-05` Docs could blur invalid-token, audience-mismatch, and insufficient-scope outcomes, weakening support clarity around `401` versus `403`.
- `T-92-06` Logout docs could overstate front-channel guarantees or collapse logout propagation URIs into post-logout redirect behavior.
- `T-92-07` DCR wording could sound like it creates a broader logout runtime rather than managing the existing metadata.

## Mitigations

- Teach the three-plug protected-route order as the canonical shipped pipeline and explain why `EnforceSenderConstraints` stays in place even for bearer-only routes.
- Pin the `401 invalid_token`, `403 insufficient_scope`, and DPoP nonce retry semantics in docs and the existing integration proof, then let Plan `92-03` add canonical contract assertions.
- Keep onboarding, operator, and DCR docs explicit about `/end_session/complete`, durable back-channel delivery, best-effort front-channel cleanup, and metadata-versus-redirect separation.
</threat_model>
