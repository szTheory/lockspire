| protocol -> storage | Active introspection resolution crosses from request-layer authorization into durable token/grant lookup. |
| durable interaction state -> host consent UI | Normalized RAR data is rendered in the host-owned consent surface before approval. |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-57-01 | I | `lib/lockspire/protocol/introspection.ex` | mitigate | Gate `authorization_details` lookup behind the existing active/client checks so inactive, revoked, reused, expired, public-client, and client-mismatched outcomes remain exactly `%{active: false}` per D-04. |
| T-57-02 | T | `lib/lockspire/protocol/introspection.ex` + storage seam | mitigate | Read normalized granted data from `ConsentGrant.authorization_details` via `token.consent_grant_id` per D-01 instead of trusting request input or token-embedded JSON. |
| T-57-03 | E | `lib/lockspire/web/controllers/introspection_controller.ex` | mitigate | Keep confidential-client authentication and thin-controller boundaries intact; only pass `consent_store: Repository` into protocol code, with no new host override seam in Phase 57. |
| T-57-04 | I | `lib/lockspire/web/live/consent_live.ex` | accept | Structural consent rendering intentionally exposes only the same normalized RAR already approved by the user, with no secrets-redaction regression or type-specific semantics added in this phase. |
| T-57-05 | S/T | `test/integration/phase43_fapi_milestone_e2e_test.exs` exercised protocol paths | mitigate | Add narrow regressions proving RAR still honors PAR-required and exact redirect-match enforcement so the new seam cannot bypass existing FAPI validation. |
