**Scenario 1 (RAR-02 + SC#1) — Host validator output is stored**

```
1. Register Application.put_env(:lockspire, :rar_validators, %{
     "payment_initiation" => Lockspire.Test.Rar.NormalizingValidator

4. Assert pushed_request.authorization_details ==
     [%{"type" => "payment_initiation", "actions" => ["initiate"], "validated" => true}]
   (NormalizingValidator's output: drops "secret", stamps "validated" => true)
5. GET /authorize?request_uri=<the URI from step 2>
6. Read back Interaction from Repository
7. Assert interaction.authorization_details == [%{"type" => "payment_initiation", "actions" => ["initiate"], "validated" => true}]
   (NOT the raw input with "secret" — D-08 chain)
```

**Scenario 2 (RAR-02 + SC#2 + D-09) — Unknown type strict-rejected**

```
1. Application.put_env(:lockspire, :rar_validators, %{})  # empty — strict reject everything
2. GET /authorize?response_type=code&...&authorization_details=[{"type":"unregistered"}]
3. Assert response is 302 redirect to redirect_uri with error=invalid_authorization_details
4. Subscribe to [:lockspire, :rar, :unknown_type] BEFORE step 2; assert telemetry event fires with %{type: "unregistered"}

**Scenario 3 (Pitfall 6 / WR-03 / SC#2) — Empty array rejected**

```
1. GET /authorize with authorization_details=[]

**Scenario 4 (Pitfall 3 / WR-04) — PAR consume does NOT double-validate**

```
1. Application.put_env(:lockspire, :rar_validators, %{
     "raises" => Lockspire.Test.Rar.RaisingValidator,
     "passthrough" => Lockspire.Test.Rar.PassthroughValidator
   })
2. POST /par with authorization_details: [{"type":"passthrough","ok":true}]
   → expect 201 with request_uri (validator runs, returns {:ok, detail})
3. NOW REPLACE Application.put_env(:lockspire, :rar_validators, %{
     "passthrough" => Lockspire.Test.Rar.RaisingValidator
   })  (point "passthrough" type at the raising validator)
4. GET /authorize?request_uri=<from step 2>
5. Assert response is 302 redirect to consent (NOT a 5xx) — proves dispatcher SHORT-CIRCUITED on PAR consume despite the now-raising validator. This proves pre_validated?: true is in effect for the PAR consume re-entry.
6. Read back Interaction; assert interaction.authorization_details unchanged from step 2's persisted output
```
