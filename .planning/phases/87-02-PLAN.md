<objective>
Teach integrators how the shipped logout propagation metadata behaves across DCR create, read, and RFC 7592 update flows without turning the guide into a hosted-product manual.
</objective>

<scope>
This plan is limited to `docs/dynamic-registration.md` per D-02, D-04, and D-05. It should add one explicit lifecycle section with concrete request/response examples and the dangerous full-replace semantics.
</scope>

<threat_model>
## Threats

- `T-87-03` Integrators may treat RFC 7592 `PUT` as patch semantics and accidentally clear or mis-rotate credentials.
- `T-87-04` DCR examples may blur logout propagation metadata with post-logout redirect URIs or imply a new logout runtime.

## Mitigations

- Add one focused create/read/update lifecycle section for `backchannel_logout_uri`, `backchannel_logout_session_required`, `frontchannel_logout_uri`, and `frontchannel_logout_session_required` per D-04.
- State plainly that RFC 7592 `PUT` is full-replace, omitted logout metadata clears prior values, returned `registration_access_token` replaces the old RAT immediately, and any rotated `client_secret` replaces the old credential immediately per D-05.
- Keep wording aligned with the canonical support page and preserve the truth model that back-channel is durable while front-channel is best-effort only per D-07 and D-08.
- Keep post-logout redirect URIs clearly separate from logout propagation URIs in both prose and examples per D-09.
</threat_model>
