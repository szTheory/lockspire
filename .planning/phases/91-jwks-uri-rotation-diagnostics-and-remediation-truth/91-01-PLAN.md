<threat_model>
## Threats

- `T-91-01` The phase could leave `private_key_jwt` and JARM with different remote-key terms or different incident-class boundaries.
- `T-91-02` The diagnostics layer could overclaim support by implying background polling, proactive readiness, or guaranteed zero-downtime rollover.
- `T-91-03` Low-level fetcher reasons could leak raw transport noise instead of the calm stable classes and safe metadata the support contract requires.
- `T-91-04` The runtime could continue emitting only protocol-local reason codes, leaving later doctor/admin/doc work to guess at the real incident state.

## Mitigations

- Add one authoritative diagnostics module that maps existing fetcher, refresh, key-selection, and signature-verification outcomes into the four stable classes plus safe supporting detail.
- Keep the normalized model explicitly bounded to the current reactive runtime: cache state, one forced refresh, last-known-good preservation, and fail-closed current request.
- Thread the normalized incident metadata through both `private_key_jwt` and JARM remote-key paths so future support surfaces consume one shared shape.
- Add direct unit tests for the diagnostics mapper plus protocol tests proving the normalized model stays aligned with current rollover and failure behavior.
</threat_model>
