<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> refresh exchange | Untrusted refresh token strings and DPoP headers attempt to rotate durable token-family state |
| refresh protocol -> repository transaction | The protocol passes a binding expectation that storage must compare atomically before mutating family state |
| repository transaction -> token family rows | A mismatch here must not create partially rotated or partially revoked family state |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-34-04 | Spoofing | `Lockspire.Protocol.RefreshExchange` | mitigate | Require a valid proof whose `jkt` matches the presented refresh token’s durable `cnf` before rotating any DPoP-bound family |
| T-34-05 | Tampering | `Lockspire.Storage.Ecto.Repository.rotate_refresh_token/6` | mitigate | Compare `expected_cnf` and persist child-token cnf inside the same `FOR UPDATE` transaction so wrong-key attempts cannot mutate family state |
| T-34-06 | Information Disclosure | refresh public error contract | mitigate | Collapse only proof-key mismatch to public `invalid_grant`, keep proof-object and proof-presentation failures on public `invalid_dpop_proof`, and retain private reason codes for telemetry and audit per D-15/D-16 |
</threat_model>
