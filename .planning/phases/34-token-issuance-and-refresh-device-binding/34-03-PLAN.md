<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| host-owned /verify -> Lockspire /token | Approval state crosses from the host seam into Lockspire issuance, but DPoP proof handling must not leak back into host-owned verification |
| public/CLI client -> device token redemption | Untrusted device clients present the winning `device_code` and DPoP proof at the same `/token` boundary |
| token exchange -> durable token persistence | The shared issuance context determines whether device-issued tokens carry durable `cnf.jkt` state |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-34-07 | Elevation of Privilege | device-code redemption path | mitigate | Bind DPoP only at the winning `/token` redemption request per D-04 so host verification approval cannot mint or pre-bind sender-constrained tokens |
| T-34-08 | Tampering | `Lockspire.Protocol.TokenExchange` device issuance branch | mitigate | Reuse the same `issuance_context` and `cnf` persistence path from auth-code issuance so device tokens cannot drift onto a separate, weaker DPoP implementation |
| T-34-09 | Repudiation | generated-host device flow proof | mitigate | Keep the `/verify` seam unchanged and prove DPoP behavior in the `/lockspire/token` exchange so support and audit traces reflect the real ownership boundary |
</threat_model>
