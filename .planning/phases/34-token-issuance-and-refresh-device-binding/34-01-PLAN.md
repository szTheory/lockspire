<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> /token controller | Untrusted DPoP headers, client credentials, and grant parameters enter Lockspire here |
| token-endpoint protocol -> replay store | Validated proofs must become durable single-use state before token issuance succeeds |
| token-endpoint protocol -> token persistence | Shared issuance context chooses whether durable cnf binding is stored on access and refresh tokens |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-34-01 | Spoofing | `Lockspire.Protocol.TokenEndpointDPoP` | mitigate | Validate `DPoP` JWTs with `Lockspire.Protocol.DPoP.validate_proof/2`, require effective DPoP clients to present a valid proof, and map failures to `invalid_dpop_proof` per D-15 |
| T-34-02 | Tampering | `lib/lockspire/protocol/token_exchange.ex` token persistence path | mitigate | Persist `cnf: %{"jkt" => proof.jkt}` on both access and refresh tokens per D-01 through the shared issuance context so later owned-surface checks do not rely on transient request state |
| T-34-03 | Repudiation | controller/protocol boundary | mitigate | Keep the controller thin and protocol-owned so DPoP policy, proof validation, and reason-code selection happen in one auditable seam rather than ad hoc header handling |
</threat_model>
