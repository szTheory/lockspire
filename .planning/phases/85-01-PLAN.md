<objective>
Allow RFC 7591 registration to accept the four logout propagation metadata fields when they are valid for Lockspire's shipped logout slice, while continuing to reject malformed or semantically invalid input as `invalid_client_metadata`.
</objective>

<scope>
This plan is limited to RFC 7591 intake validation and normalization. It does not add RFC 7592 write/update semantics; those remain Phase 86 work.
</scope>

<threat_model>
## Threats

- `T-85-01` Invalid or ambiguous logout metadata could reach durable client state and later drive incorrect back-channel delivery or misleading front-channel behavior.
- `T-85-02` Relaxed parsing could silently coerce malformed values, causing self-service clients to believe metadata was accepted when Lockspire actually changed it.

## Mitigations

- Accept only explicit, validated URI and boolean shapes.
- Preserve RFC-shaped `invalid_client_metadata` failures with field-specific reasons.
- Keep front-channel origin checks aligned with the already-shipped operator workflow.
</threat_model>

<tasks>
- [ ] `85-01-01` Replace the blanket `validate_unsupported_logout_metadata/1` rejection with a shared, explicit logout metadata validator for DCR wire input.
  - Files: `lib/lockspire/protocol/registration.ex`
  - Action: introduce a focused helper that validates the four logout fields, trims URI strings, enforces absolute `http`/`https` URIs without fragments, requires the paired URI when `*_session_required` is true, and checks `frontchannel_logout_uri` origin against `redirect_uris`.
  - Verify: negative tests fail with `%Registration.Error{code: :invalid_client_metadata, field: ..., reason: ...}`.

- [ ] `85-01-02` Reuse existing client/logout primitives instead of inventing a second truth model.
  - Files: `lib/lockspire/clients.ex`, `lib/lockspire/admin/clients.ex`, `lib/lockspire/protocol/registration.ex`
  - Action: rely on `Lockspire.Clients.validate_logout_uri/1` and `frontchannel_logout_origin_matches_redirect_uri?/2` from the new DCR helper; avoid copying URI semantics into ad hoc DCR-only code.
  - Verify: no new conflicting validation rules are introduced for the same four fields.

- [ ] `85-01-03` Extend fixtures and protocol tests to cover positive and negative RFC 7591 inputs.
  - Files: `test/support/fixtures/dcr_fixtures.ex`, `test/lockspire/protocol/registration_test.exs`
  - Action: add fixture builders for valid logout metadata, invalid boolean values, missing URI with `*_session_required`, malformed URI, and front-channel origin mismatch.
  - Verify: `mix test test/lockspire/protocol/registration_test.exs`.
</tasks>

<success_criteria>
- RFC 7591 registration no longer rejects logout metadata solely because the fields are present.
- Valid logout propagation metadata is accepted under the shipped Lockspire truth model.
- Invalid or semantically incoherent logout metadata fails as `invalid_client_metadata` with stable field attribution.
- No RFC 7592 write behavior is widened in this plan.
</success_criteria>
