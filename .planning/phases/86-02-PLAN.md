<objective>
Preserve the existing RFC 7592 lifecycle guarantees while logout metadata becomes writable: rotated RATs stay single-use, update responses reflect persisted truth, and provenance/audit behavior remains stable.
</objective>

<scope>
This plan covers management-update lifecycle invariants and their proof. It does not add new admin UI features or new logout behavior outside stored metadata management.
</scope>

<threat_model>
## Threats

- `T-86-03` Adding logout field persistence could accidentally break registration access token rotation or reuse prevention.
- `T-86-04` Self-service update responses could diverge from the persisted client or operator-visible truth.
- `T-86-05` Update persistence could regress provenance or audit guarantees even if the functional update succeeds.

## Mitigations

- Prove that every successful logout metadata update rotates the RAT and invalidates the old token immediately.
- Assert update responses serialize the same persisted logout metadata that read/operator paths will later expose.
- Verify provenance and update audit behavior remain unchanged for self-registered clients.
</threat_model>

</success_criteria>
