<objective>
Add a single `validate_rfc9068_compliance/2` step inside `Lockspire.Plug.VerifyToken` that runs after `JOSE.JWT.verify_strict/3` succeeds and before `apply_restrictions/2`, enforcing the five RFC 9068 / RFC 8725 compliance rules currently missing. Each check emits a distinct atom `reason_code` through the structured error map shape (D-04), so adopters who break a rule see exactly which RFC 9068 / RFC 8725 clause was violated in the `WWW-Authenticate: ... error_description="..."` header.

Per D-02, the five checks run in this order, each producing its named atom on failure:
1. JWT protected-header `typ` is a case-insensitive variant of `at+jwt` (or `application/at+jwt`, per D-03 normalization) → on failure `:invalid_typ`
2. `claims["iss"]` exact-string-equals `Lockspire.Config.issuer!/0` → on failure `:invalid_issuer`
3. `claims["exp"]` is a positive integer → on failure `:missing_exp`
4. `claims["iat"]` is a positive integer → on failure `:missing_iat`
5. `claims["sub"]` is a non-empty string → on failure `:missing_sub`

Per CONTEXT.md `<deferred>` line 155, this plan also DELETES the now-obsolete comment `# Missing exp is currently treated as valid unless stricter policy requires it.` at `verify_token.ex:366` — once `:missing_exp` is an enforced reason code, the comment is wrong.

Purpose: An adopter who sends a JWT lacking RFC 9068 mandatory claims, or with `typ: JWT` instead of `typ: at+jwt`, or with an `iss` that doesn't match the configured Lockspire issuer, sees a distinct named rejection — not a generic 401, and not a `:invalid_signature` confusion. Phase 98 Success Criterion #2 (ROADMAP.md line 67) is met.

Output: Modified `verify_token.ex` with one new private function (`validate_rfc9068_compliance/2`), five new structured-error helpers (one per reason code), one new private helper to normalize and compare the `typ` header per D-03, the new step wired into `verify_signature_and_claims/2`, and the obsolete comment deleted. Modified test file with at least 10 new test cases (two per reason code: the rejection case and a "well-formed token still accepted" reference case for that field).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Verified JWT bytes → `validate_rfc9068_compliance/2` | The JWT signature has been verified by `JOSE.JWT.verify_strict/3` before this step runs. The claims map is trustworthy in the sense that it was signed by a key in the configured JWKS; the question this step answers is whether the trustworthy claims conform to RFC 9068's mandatory shape. |
| `Lockspire.Config.issuer!/0` → in-process literal compare | The configured issuer string is a deployment-time invariant. D-02 step 2 does exact-string-compare against `claims["iss"]`; this is the canonical issuer-pinning gesture per RFC 9068 §4. |
| Error map → operator log / WWW-Authenticate header | The `error_description` string is emitted verbatim into the operator log AND the response header. Must be printable ASCII and free of unescaped double quotes that would break the header value. |
