<objective>
Derive `challenge:` on every `VerifyToken`-produced structured error map from the token's binding claim (via `binding_type/1` at `verify_token.ex:288-300`), with the request's `Authorization` scheme as a tiebreaker when the token has no `cnf` claim. This replaces the four hard-coded `challenge: :bearer` sites in `verify_token.ex` (at lines 187, 198 — `invalid_audience_error/2` and `insufficient_scope_error/1`) AND the new RFC 9068 error sites from Plan 03 AND the Plan 01 opaque-token error site AND the implicit `:bearer` defaults in `require_token.ex` normalizers (lines 81, 99, 113).

Per D-05 the mapping is:
1. Token has `cnf.jkt` → `challenge: :dpop`
2. Token has only `cnf["x5t#S256"]` → `challenge: :bearer` (RFC 8705 §3: mTLS reuses Bearer; RFC 9449 §7.1 only redefines the scheme for DPoP)
3. No `cnf` claim AND request used `Authorization: DPoP <...>` → `challenge: :dpop` (rare misconfigured-client path; still a DPoP-shaped failure for protocol fidelity)
4. Else → `challenge: :bearer`

Per D-06 this is WIRE-UP, not a new mechanism: the downstream emission path is unchanged. `ProtectedResourceChallenge.put_dpop_challenge/2` at `web/protected_resource_challenge.ex:37-44` already emits the RFC 9449-compliant `DPoP realm="..." error="..." error_description="..." algs="..."` header when `challenge: :dpop` reaches `require_token.ex:51`. Plan 04 just changes WHICH errors carry `challenge: :dpop`.

Purpose: An adopter who sends a DPoP-bound `at+jwt` and fails audience or scope checks gets `WWW-Authenticate: DPoP ...` — not `Bearer ...` — because the protocol scheme should match the token's binding type per RFC 9449 §7.1. Conversely, an adopter who sends an mTLS-bound `at+jwt` and fails the same checks still gets `WWW-Authenticate: Bearer ...` because RFC 8705 §3 specifies the Bearer scheme for mTLS-bound tokens. The four hard-coded `:bearer` sites and the implicit normalizer defaults converge to a single derivation point.

Output: Modified `verify_token.ex` with a new `challenge_for/2` (or similar — executor's discretion on name) helper that takes the claims map AND the authorization scheme and returns one of `:bearer | :dpop`. Modified error helpers (`invalid_audience_error/2`, `insufficient_scope_error/1`, the new RFC 9068 `rfc9068_error/1` from Plan 03, and the opaque-token error from Plan 01) to call this helper. Plan 01's opaque-token path may continue hard-coded to `:bearer` since the token has no binding (no cnf can be parsed — it's opaque) and the request's authorization scheme may legitimately tiebreak — executor's choice on whether to thread scheme into the opaque-token helper. Modified `require_token.ex` normalizers' fallthrough behavior to pass through any explicitly-set `challenge:` from the structured map (existing behavior already does this via `Map.get(error, :challenge, :bearer)` — Plan 04 confirms this and adjusts call-sites that bypassed it). New tests in `verify_token_test.exs` and `require_token_test.exs` for the four binding-derivation paths.
</objective>

  has_dpop? = present?(Map.get(cnf, "jkt"))

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Verified claims map → `challenge_for/2` | The claims have been signature-verified upstream. The `cnf` claim, if present, reflects the binding the IdP issued the token under. Phase 98 trusts the signed claim — it does NOT cross-check against the request's actual DPoP proof / mTLS cert (that's `EnforceSenderConstraints`' job). |
| Request `Authorization:` scheme → `challenge_for/2` tiebreaker | Untrusted client input. The scheme is the literal `"Bearer"` or `"DPoP"` token from the header. An attacker can pick either freely. The tiebreaker is invoked ONLY when the token carries no binding claim — its purpose is protocol fidelity in the error response, not authentication. |
| `WWW-Authenticate` header value → adopter / observability | The scheme letter (`DPoP` vs `Bearer`) is the strongest channel by which a misconfigured client learns what shape the server expected. Wrong scheme leaks no secrets; right scheme accelerates correct adopter remediation. |
