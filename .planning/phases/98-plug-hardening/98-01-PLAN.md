<objective>
Add a front-edge structural opaque-token check inside `Lockspire.Plug.VerifyToken.verify_token/3` that rejects any token whose trimmed form does not split into exactly three non-empty Base64URL segments by `.`. The rejection emits a structured error map carrying `reason_code: :opaque_token_not_accepted`, `challenge: :bearer`, `error: "invalid_token"`, and a distinct `error_description: "opaque tokens not accepted on this route"`. This closes VERIFIER-01 by ending the silent `:malformed` lumping that today swallows opaque tokens at `extract_kid/1`'s rescue clause.

Purpose: An adopter who accidentally sends a `/userinfo`-style stored opaque token (32 bytes, Base64URL, no dots) to a host Phoenix API route protected by `VerifyToken` must see a distinct named rejection, not be told the token was "malformed" alongside genuine JWT corruption. Per D-01 the detection is structural — three-segment shape — not Base64URL-payload header peek, so it does not waste a JOSE parse on tokens that are obviously not JWTs.

Output: Modified `verify_token.ex` with the new check inserted ahead of `extract_kid/1` inside `verify_token/3`; modified test file with an opaque-token fixture and assertions for the new path plus a regression assertion that "Bearer not.a.jwt" still classifies as `:malformed`.
</objective>

| T-98-01-03 | Tampering | A client could craft a token whose Base64URL characters are valid for the regex but whose decoded bytes are garbage, hoping to bypass the structural check while still being rejected by JOSE | accept | Same disposition as T-98-01-01: the structural check is a fast-path classifier, not a signature/integrity check.
