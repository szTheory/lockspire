<objective>
Close VERIFIER-06 with both mechanisms allowed by the OR-clause in REQUIREMENTS.md:24-29 (D-07 ships both for defense-in-depth):

1. **Option mechanism:** Add `enforce_audience: true | false` (default `false`) to `VerifyToken`'s `NimbleOptions` schema. When `enforce_audience: true` is set and neither `:audience` nor `:audiences` is supplied, `init/1` raises `ArgumentError`.

Per D-08, the install template's commented `:lockspire_protected_api` pipeline retains `enforce_audience: true` so new adopters who later uncomment the block and delete the `audience:` line get the loud `init/1` raise.

Purpose: An adopter who mounts `Lockspire.Plug.VerifyToken` without an audience on a pipeline that declares `enforce_audience: true` (the install-template default per D-08) sees the plug fail loudly at compile/boot time, not silently accept cross-API tokens at runtime. Cross-API token reuse — a documented attack class — is structurally closed on the blessed path.

Output: Modified `verify_token.ex` with one new schema key and one new validation; new tests for the raise; new contract-test clause that fails loudly if any of the four canonical-pipeline declarations omits `audience:`.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Adopter mounts `Lockspire.Plug.VerifyToken` in their host's router pipeline | The adopter chooses whether to supply `audience:`. Without enforcement, a missing audience silently allows cross-API token reuse — a documented OAuth attack class (RFC 9068 §5.5, RFC 8725 §3.12). |
| `priv/templates/lockspire.install/router.ex` → generated host project | The install template is the default scaffold new adopters inherit. Whatever the template ships becomes the default deployed shape across the adopter base. |
| Contract test → CI | The four-file canonical-pipeline content-hash test gates merges. If the new audience-substring clause is bypassable, the contract is hollow. |
