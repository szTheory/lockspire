**Mapped:** 2026-05-06
**Scope:** Install Mix tasks, generator/template inventory, no-overwrite semantics, generated-host fixtures, and install-proof tests.

## Reusable Patterns

### Mix task shape

Use the same CLI structure as `lockspire.install` and `lockspire.client.create`:

- `OptionParser.parse/2` with `strict: [...]`
- `invalid != []` raises immediately
- `--help` prints a heredoc
- the actual operation lives in a dedicated library module rather than the task module

### No-overwrite generator baseline

`Lockspire.Generators.Install.ensure_file!/2` already provides the core safety behavior:

- unchanged file => print `* unchanged ...`
- modified existing file => raise `Refusing to overwrite modified file`
- missing file => create it

Phase 63 should preserve this for host-owned files and refine it only for managed scaffolding under manifest control.

### Runtime validation posture

`Lockspire.Config` and `Lockspire.Oban` establish the repo's current diagnostics tone:

- fail fast
- name the missing key or invalid config surface directly
- do not hide failure behind indirect later behavior

`mix lockspire.verify` should reuse that tone, but as a post-install operator command rather than a boot-time replacement.

### Generated-host proof analog

`test/support/generated_host_app_web/router.ex` already shows the truthful proof target:

- host-owned `/verify` routes in the host router
- host login/session routes in the same router
- `forward "/lockspire", Lockspire.Web.Router` mounted through the host router

Phase 63 proof should move toward this surface so the canonical install claim is actually about a host app, not just the embedded router in isolation.

## Recommended New Patterns

### Template inventory metadata

Extend `Lockspire.Generators.Templates.all/0` entries to include:

- `ownership: :managed | :host_owned`
- `manifest_key`
- `upgradeable?: true | false`

That keeps the artifact classification close to the generated file inventory instead of scattering it across separate allowlists.

### Verify check result shape

Use one normalized result struct or map per diagnostics check:

- `id`
- `status`
- `summary`
- `details`
- `fix`

That allows `mix lockspire.verify` to print stable messages while the checker logic stays testable in library modules.

### Managed-upgrade manifest

Use one repo-local manifest file generated by install, containing at minimum:

- generator version or template version
- managed file paths
- rendered checksum per managed file
- mount path or other generation inputs needed for safe comparison

Upgrade should act only on manifest-tracked managed files and should never infer host-owned ownership from path heuristics alone once a manifest exists.
