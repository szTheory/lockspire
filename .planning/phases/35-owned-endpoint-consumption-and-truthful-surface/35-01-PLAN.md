<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> `/userinfo` | Untrusted authorization scheme, opaque access token, and DPoP proof enter Lockspire here |
| userinfo protocol -> replay store | Accepted proofs must become durable single-use state before claims are returned |
| protected-resource validator -> token persistence | Validated proof keys must match durable `cnf.jkt`, not mutable client policy or request-local assumptions |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-35-01 | Spoofing | `Lockspire.Protocol.ProtectedResourceDPoP` | mitigate | Require `Authorization: DPoP`, validate proof signature/claims, require `ath`, and compare proof `jkt` to persisted `cnf["jkt"]` per D-01, D-07, and D-08 |
| T-35-02 | Tampering | replay-state + `userinfo` proof use | mitigate | Reuse durable replay recording so the same proof cannot be replayed across nodes or restarts per D-06 |
| T-35-03 | Elevation of Privilege | `userinfo` controller/protocol seam | mitigate | Keep the controller thin and centralize DPoP correctness in protocol modules so no endpoint-local downgrade or bypass path appears per D-05 |
</threat_model>
