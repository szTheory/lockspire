## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | #F5F7F8 | Page background and low-emphasis shell background |
| Secondary (30%) | #FFFFFF | Verification card, review panel, inline notice surfaces |
| Accent (10%) | #0F766E | Primary action button, focus ring, confirmed code chip, active step emphasis only |
| Destructive | #B42318 | Deny action, destructive warning copy, destructive focus state only |

Accent reserved for: `Review device request`, `Approve device`, the visible user-code confirmation chip, and the current-step focus ring. Do not use accent on every link, border, or passive label.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Review device request |
| Empty state heading | Enter your device code |
| Empty state body | Type the code shown on your device. You will review the request details before anything is approved. |
| Error state | We couldn't verify that code. Check the code on your device and try again, or restart the sign-in flow to get a new code. |
| Destructive confirmation | Deny request: Only deny if the code or app does not match your device. Denying ends this request and the device must start again. |

## Interaction Contract

| State | Required UI | Interaction Rule |
|------|-------------|------------------|
| Initial `GET /verify` | Single centered card with heading, short explanation, code input, and helper text | If `verification_uri_complete` is present, prefill the code field only. Do not auto-submit, auto-look-up, or mutate state on page load. |
| Code entry | Visible editable `user_code` field with normalization-friendly presentation | Submission must require an explicit user click on `Review device request`. GET stays side-effect free. |
| Review step | Show the normalized `user_code`, client name, requested scopes, signed-in account context, and a short possession-check prompt | The page must explicitly tell the user to confirm the code matches the requesting device before approval. |
| Approval action | Primary `Approve device` button and secondary destructive `Deny request` button | Approval and denial must be separate explicit submits on the opaque verification handle, never on the raw `user_code` alone. |
| Invalid or expired code | Neutral message surface with no distinction between nonexistent and expired codes | Collapse `:not_found` and `:expired` into the same host-facing copy to avoid an existence oracle. |
| Not active / already used | Neutral message surface plus restart guidance | Do not expose internal lifecycle names. Tell the user to restart from the device if the request is no longer active. |
| Rate-limited lookup or submit | Inline error banner above the form plus preserved form state where safe | Use 429 semantics with short retry guidance. Do not reveal whether the submitted code exists. |

Required layout:

- Use a single-column shell with a maximum readable width of 480px.
- Keep the code confirmation prompt above the approve and deny actions.
- Stack approve and deny actions vertically on mobile and keep at least 16px between them.
- Keep approval and denial in the same visual group so the risk tradeoff is obvious before submit.

Required visual hierarchy:

- Render the user code in display style inside a bordered chip or panel before approval.
- Keep client name and requested scopes above the action buttons.
- Keep rate-limit, invalid-code, and destructive notices in secondary surfaces; never as modal overlays in the starter seam.

Generated seam notes:

- Generated comments in host-owned routes, controller or LiveView files, and docs must state that `verification_uri_complete` is a prefill optimization only.
- Generated comments must warn against logging raw query strings or raw user codes.
- Generated comments must point the host to device-flow rate-limit guidance for both `GET /verify` and `POST /verify`.
- Generated examples must show explicit actor binding at approval time and must never imply approval via GET.
