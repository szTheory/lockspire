<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Host browser seam -> verification protocol | Untrusted user-entered `user_code` crosses into typed lookup semantics. |
| Verification protocol -> repository | Approval and denial cross into durable state transitions that must resist concurrent mutation. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-31-01 | Tampering | `Lockspire.Storage.Ecto.Repository.transition_device_authorization/3` | mitigate | Use `FOR UPDATE` plus expected-state checks so approve/deny/consume cannot win twice during double-submit or poll races. |
| T-31-02 | Information Disclosure | `Lockspire.Protocol.DeviceVerification` | mitigate | Normalize and hash the `user_code` before storage lookup, return typed internal outcomes without exposing raw-code identifiers, and keep post-lookup mutations keyed by `verification_handle` instead of `user_code`. |
| T-31-03 | Spoofing | `Lockspire.Protocol.DeviceAuthorization` | mitigate | Emit `verification_uri_complete` only as a prefill URL field; do not couple it to any lookup or approval side effects in protocol code. |
</threat_model>
