# Milestone v1.14: Advanced Authorization & Resource Targetting

**Status:** ✅ SHIPPED 2026-05-06
**Phases:** 54-58
**Total Plans:** 12

## Overview

v1.14 delivered OAuth 2.0 Resource Indicators (RFC 8707) and Rich Authorization Requests (RFC 9396) as a coherent advanced-authorization slice. Lockspire can now accept and validate targeted resource requests, ingest and normalize rich authorization payloads, persist them durably on consent grants, expose them to Resource Servers through introspection, and advertise the shipped surface truthfully in discovery and documentation.

## Phases

### Phase 54: Resource Indicators (RFC 8707)
**Goal**: Users and clients can request tokens targeted at specific Resource Servers.
**Depends on**: None (v1.13 completed)
**Requirements**: RES-01, RES-02, RES-03
**Plans**: 1 plans

Plans:

- [x] 54-01: Resource parameter validation, targeted audience claims, and refresh-token downscoping

**Details:**
Resource Indicators support landed on the existing authorization-code and refresh-token exchanges. The auth request pipeline now validates absolute `resource` URIs without fragments, carries the granted resource set through authorization state, and narrows issued and rotated access tokens to the requested granted audience subset. Phase 54 also established the integration-test harness later reused by the RAR phases.

### Phase 55: RAR Protocol Intake
**Goal**: Clients can submit structured authorization details via PAR and Authorization requests.
**Depends on**: Phase 54
**Requirements**: RAR-01
**Plans**: 3 plans

Plans:

- [x] 55-01: Database and domain extension for RAR intake
- [x] 55-02: Protocol validation, parsing, and PAR persistence logic
- [x] 55-03: End-to-end integration tests for RAR intake

**Details:**
Phase 55 introduced `authorization_details` transport support without widening the host seam. PAR and direct authorization requests now parse RFC 9396 payloads, enforce a direct-request size cap, and persist the decoded details through PAR and interaction state so later validation and introspection phases can build on a durable, normalized foundation.

### Phase 56: RAR Domain Validation & Storage
**Goal**: Host apps can define and validate custom RAR types using idiomatic Elixir patterns.
**Depends on**: Phase 55
**Requirements**: RAR-02, RAR-03
**Plans**: 6 plans
**UI hint**: yes

Plans:

- [x] 56-01: Scaffolding, validator fakes, retrofit inventory
- [x] 56-02: Behaviour, config, error formatting, fingerprinting
- [x] 56-03: Validator dispatcher and telemetry
- [x] 56-04: Durable consent/token storage
- [x] 56-05: Protocol integration and consent reuse fingerprinting
- [x] 56-06: E2E verification, retrofit, and full-suite stabilization

**Details:**
Phase 56 added the host-owned validator seam, validator dispatch/telemetry, normalized RAR fingerprinting, durable consent-grant persistence, and `consent_grant_id` linkage on issued tokens. It also enforced empty-array rejection and remembered-consent reuse by RAR fingerprint, proving that rich authorization stays durable and replay-safe across issuance and refresh rotation.

### Phase 57: RAR Introspection & Verification
**Goal**: Resource Servers can retrieve rich authorization details via introspection.
**Depends on**: Phase 56
**Requirements**: RAR-04, V-01, V-02
**Plans**: 1 plans

Plans:

- [x] 57-01: Grant-backed RAR introspection, structural consent proof, and narrow verification/regressions

**Details:**
Phase 57 resolved RAR details by grant reference during active introspection, kept tokens compact by avoiding token-embedded RAR JSON, and proved that the host-owned consent surface renders the same normalized payload structurally. It also added narrow RAR-aware FAPI regressions to ensure PAR-required posture and exact redirect matching still hold.

### Phase 58: Milestone Closure & Discovery
**Goal**: The system truthfully advertises its new capabilities and provides clear integrator guidance.
**Depends on**: Phase 57
**Requirements**: META-01, META-02, DOC-01
**Plans**: 1 plans

Plans:

- [x] 58-01: Truthful discovery metadata, custom RAR host guide, and release-contract closure handoff

**Details:**
Phase 58 published truthful `resource_indicators_supported` and `authorization_details_types_supported` discovery metadata, added an executable host-owned RAR consent guide, and aligned README/HexDocs/release-contract checks to the shipped surface.

## Milestone Summary

**Decimal Phases:**

- None

**Key Decisions:**

- Selected Resource Indicators and Rich Authorization Requests as the next leverage-heavy trust wedge for complex domain integrations.
- Preserved the host-owned seam by keeping RAR validation, consent semantics, and rendering customizable through behaviors and generated host code.
- Kept introspection compact-by-reference by resolving RAR details through `consent_grant_id` instead of embedding full RAR JSON into tokens.

**Issues Resolved:**

- Added exact-target resource validation and audience downscoping to prevent over-broad token audiences.
- Added durable RAR normalization, fingerprinting, and remembered-consent protection.
- Added truthful discovery and support-contract documentation for the shipped advanced authorization surface.

**Issues Deferred:**

- Historical Phase 37 verification debt remains acknowledged outside this milestone close and is tracked in `STATE.md`.

**Technical Debt Incurred:**

- Phase 56/57 Nyquist validation bookkeeping was not fully reconciled even though shipped verification is complete.
