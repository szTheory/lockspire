# v1.21 Resource Server (API Protection) Requirements

## Core Validation Plug (VAL-PLUG)
- **VAL-PLUG-01**: Lockspire MUST provide a standard Plug (e.g., `Lockspire.Plug.VerifyToken`) to easily protect host Phoenix API routes.
- **VAL-PLUG-02**: The Plug MUST validate JWT signatures using Lockspire's registered/issued JWKS.
- **VAL-PLUG-03**: The Plug MUST enforce token expiration (`exp`) and not-before (`nbf`) standard claims.
- **VAL-PLUG-04**: The Plug MUST optionally enforce required scopes or audience (`aud`) restrictions passed via Plug options.

## Sender-Constraining & Cryptographic Binding (VAL-BIND)
- **VAL-BIND-01**: If an access token contains a DPoP `cnf` claim, the Plug MUST transparently extract and validate the DPoP proof header against the token's confirmation claim.
- **VAL-BIND-02**: If an access token contains an MTLS `x5t#S256` `cnf` claim, the Plug MUST transparently extract the client certificate (using `Lockspire.MTLS.Extractor`) and validate the thumbprint.
- **VAL-BIND-03**: The Plug MUST reject requests with a 401 Unauthorized if cryptographic binding validation fails.

## Developer Ergonomics & Error Handling (VAL-DX)
- **VAL-DX-01**: Upon successful validation, the Plug MUST assign the validated token, the associated `client_id`, and extracted claims (like `sub`) into `conn.assigns` for easy consumption by host controllers.
- **VAL-DX-02**: The Plug MUST emit standardized HTTP 401 Unauthorized responses on failure, including the `WWW-Authenticate` header formatted according to RFC 6750 (Bearer) or RFC 9439 (DPoP), complete with `error` and `error_description`.
- **VAL-DX-03**: The Plug MUST provide actionable, developer-friendly error messages in the logs to distinguish between missing tokens, expired tokens, signature failures, and binding mismatches.
