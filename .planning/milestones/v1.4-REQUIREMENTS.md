# Requirements Archive: v1.4

**Archived:** 2026-04-26
**Milestone:** v1.4
**Release posture at archive:** Planning milestone complete; package version `0.2.0` remains preview, and the shipped surface now includes the verified JAR slice plus the earlier PAR and PAR-policy work.

## v1.4 Requirements

### JAR Foundation

- [x] **JAR-01**: Clients can send signed request objects by value or through PAR-backed request-object flow on the existing authorization code + PKCE path.
- [x] **JAR-02**: Lockspire validates request object signatures using the client’s key material.
- [x] **JAR-03**: Lockspire enforces the required request-object claims and request-object freshness rules before host login logic resumes.

### JAR Operator Controls

- [x] **JAR-05**: Discovery metadata advertises the shipped JAR support truthfully.
- [x] **JAR-06**: Operators can configure and inspect JAR enforcement policy through the existing admin surface.

### Deferred Scope

- [ ] **JAR-04**: Support JAR decryption (optional/future)

## Requirement Outcomes

| Requirement | Phase | Final Status | Notes |
|-------------|-------|--------------|-------|
| JAR-01 | Phase 22 | Complete | JAR request objects now compose through `/authorize`, `/par`, consent, and `/token` without changing the downstream auth-code path. |
| JAR-02 | Phase 21/22 | Complete | Signature verification uses client keys and is exercised by the JAR protocol and integration proofs. |
| JAR-03 | Phase 21/22 | Complete | Mandatory claim validation and freshness ceilings stay inside the request-object seam. |
| JAR-05 | Phase 23 | Complete | Discovery metadata advertises only the shipped JAR slice. |
| JAR-06 | Phase 23 | Complete | Operator policy, persistence, and admin surfaces cover the shipped JAR enforcement controls. |
| JAR-04 | Deferred | Deferred | JAR decryption was intentionally not shipped in v1.4. |

## Deferred to Next Milestone Planning

### Future Requirements

- **JAR-04**: Clients can support JAR decryption if the next milestone chooses to expand the request-object surface.
- **DCR-01**: Developers can self-register OAuth clients through dynamic client registration.
- **DEVICE-01**: Devices can complete OAuth authorization through the device authorization grant.
- **SCON-01**: Clients can use stronger sender-constrained token modes beyond the baseline bearer-token preview surface.
