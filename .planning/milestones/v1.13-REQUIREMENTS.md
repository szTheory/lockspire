# v1.13 OpenID Connect CIBA Requirements

## CIBA Protocol Core
- **CIBA-01 (BC-Authorize Endpoint):** The system must expose a backchannel authentication endpoint (`/bc-authorize`) that validates RP requests, issues an `auth_req_id`, and defines the `interval` and `expires_in`.
- **CIBA-02 (Token Grant Type):** The `/token` endpoint must accept `grant_type=urn:openid:params:grant-type:ciba` and validate the `auth_req_id` against the durable storage.
- **CIBA-03 (Poll Mode Logic):** The system must track client polling frequency. If a client polls too quickly, it must return `slow_down`. If consent is unresolved, it must return `authorization_pending`.

## Host Integration & Anti-Spam
- **CIBA-04 (Host Delivery Behaviour):** The system must define an Elixir Behaviour (e.g., `Lockspire.Host.ciba_notify/1`) that allows the host application to securely resolve the user and trigger the out-of-band notification (e.g., smartphone push).
- **CIBA-05 (Anti-Spam & User Code):** The system must strictly enforce `user_code` and `binding_message` matching, delegating hint resolution (`login_hint_token`, `login_hint`, `id_token_hint`) to the host to prevent RP-initiated DoS/MFA fatigue attacks.

## Advanced Delivery Modes
- **CIBA-06 (Ping Mode):** The system must support the `ping` delivery mode, utilizing Oban to dispatch a webhook with the `auth_req_id` to the RP's `client_notification_endpoint` when consent is resolved.
- **CIBA-07 (Push Mode):** The system must support the `push` delivery mode, utilizing Oban to dispatch a webhook containing the actual access and ID tokens to the RP's `client_notification_endpoint`.
