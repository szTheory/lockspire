# v1.21 Resource Server (API Protection) Context

## Phase 79: Core Validation Plug Design Decisions

During the discuss phase, we established the following architectural invariants for the `Lockspire.Plug.VerifyToken` and related components:

1. **Failure Mode: The "Two-Plug" Pattern**
   - **Decision:** Separate token extraction/verification from authorization enforcement.
   - **Details:** `Lockspire.Plug.VerifyToken` acts as a "soft" plug. It extracts the token, verifies the signature, and evaluates time-based claims, but **never halts** the connection. Instead, it assigns the result or specific error state. `Lockspire.Plug.RequireToken` acts as the "strict" enforcer, checking the assigned state and halting with `401 Unauthorized` and `WWW-Authenticate: Bearer` if necessary.
   - **Rationale:** Preserves pipeline composability for endpoints where authentication is optional, mirroring idiomatic Elixir auth libraries (`phx.gen.auth`, `Guardian`).

2. **State Encapsulation: The `%Lockspire.AccessToken{}` Struct**
   - **Decision:** Use a formal struct to represent the token lifecycle in the request.
   - **Details:** 
     ```elixir
     %Lockspire.AccessToken{
       token: String.t() | nil,
       claims: map() | nil,
       client_id: String.t() | nil,
       binding_type: :bearer | :dpop | :mtls | nil,
       binding_thumbprint: String.t() | nil,
       error: atom() | nil
     }
     ```
   - **Rationale:** Prevents primitive obsession with raw maps, providing a clean, pattern-matchable contract for controllers and setting the stage for Phase 80 (DPoP/MTLS bindings).

3. **Key Resolution: Bypassing the Database**
   - **Decision:** Introduce a lightweight ETS cache layer for JWKS resolution.
   - **Details:** Avoid calling Ecto on every HTTP request. Introduce a `Lockspire.KeyCache` GenServer that fetches active signing keys on boot and refreshes them periodically, storing them in a `read_concurrency: true` ETS table. `VerifyToken` reads exclusively from ETS.
   - **Rationale:** Protects the database connection pool and ensures microsecond key resolution latency for high-volume resource servers.
