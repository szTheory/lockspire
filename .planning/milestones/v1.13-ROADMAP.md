## Phase Details

### Phase 51: Core Protocol & Poll Mode
**Goal**: The server can receive CIBA authorization requests and issue tokens via the polling grant type.
**Depends on**: Phase 50
**Requirements**: CIBA-01, CIBA-02, CIBA-03
**Success Criteria** (what must be TRUE):
  1. A client can submit a valid request to the new `/bc-authorize` endpoint and receive an `auth_req_id` and `expires_in`.
  2. The `/token` endpoint accepts `grant_type=urn:openid:params:grant-type:ciba`.
  3. The server correctly enforces polling intervals and returns `authorization_pending` or `slow_down` when appropriate.

**Plans:** 3 plans
- [x] 51-01-PLAN.md — Establish durable storage foundation and domain models for CIBA authorizations.
- [x] 51-02-PLAN.md — Implement /bc-authorize endpoint, validation protocol, and discovery metadata.
- [x] 51-03-PLAN.md — Integrate CIBA grant type into token exchange and implement polling state machine.

### Phase 52: Host Delegation & Notification Seams
**Goal**: The protocol state is connected to the host application's out-of-band notification and consent mechanisms.
**Depends on**: Phase 51
**Requirements**: CIBA-04, CIBA-05
**Success Criteria** (what must be TRUE):
  1. A new Elixir Behaviour is defined for the host app to handle incoming CIBA requests (e.g., to trigger a push notification).
  2. The host app has a secure seam to report the user's consent (approve/deny) asynchronously back to Lockspire.
  3. Lockspire accurately validates the `login_hint_token`, `user_code`, and `binding_message` to prevent unsolicited spam.
**Plans**:
- [x] 52-01-PLAN.md — Connect CIBA protocol state to host behaviours and implement async approval API.

### Phase 53: Ping & Push Delivery Modes
**Goal**: Lockspire reliably delivers authentication results to Relying Parties via webhooks.
**Depends on**: Phase 52
**Requirements**: CIBA-06, CIBA-07
**Success Criteria** (what must be TRUE):
  1. Clients registered with `client_notification_endpoint` receive webhooks upon user consent resolution.
  2. Lockspire uses Oban for resilient, retriable webhook delivery to RP endpoints.
  3. Push mode securely delivers the tokens directly to the RP endpoint without requiring a follow-up token request.
