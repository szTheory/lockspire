# Requirements Archive: v1.28 Admin UI Operator Experience Polish

**Status:** SHIPPED 2026-06-03
**Milestone:** v1.28 Admin UI Operator Experience Polish
**Defined:** 2026-06-03
**Core Value:** A Phoenix team can become a trustworthy OAuth/OIDC provider inside its existing app without inventing the dangerous parts itself.

**Milestone Goal:** Make the Lockspire admin UI feel like one coherent operator product, not a set of accumulated LiveViews. Preserve the current embedded-library boundary and protocol behavior while tightening information architecture, reusable admin components, page hierarchy, mobile behavior, action safety, demo seed expressiveness, and screenshot-driven verification.

**Design Decision:** Double down on the existing BEM/design-token CSS direction introduced by the previous polish pass. Do not switch to Tailwind or introduce a theming engine. Treat the current uncommitted overview/nav/security/DCR/demo-seed work as the baseline for v1.28.

## Requirements

### Operator Journey + IA

- [x] **UX-IA-01**: The admin UI names and supports the main operator jobs-to-be-done: platform setup proof, partner/client onboarding, security posture review, support incident investigation, and runtime operations triage.
- [x] **UX-IA-02**: `/admin` remains the default cockpit and routes each major operator need to the right next screen with specific, action-oriented copy.
- [x] **UX-IA-03**: Navigation and secondary navigation expose top-level tasks, not every resource; pages must include local links where cross-object journeys matter.

### Design System + CSS Architecture

- [x] **DS-01**: Admin UI primitives are centralized in reusable components/classes for buttons, action bars, alerts, confirmation panels, forms, fields, description lists, summary stats, resource rows, badges, and code/value blocks.
- [x] **DS-02**: Polished admin screens remove ad hoc inline layout/color styles in favor of stable `lockspire-admin-*` BEM-style classes.
- [x] **DS-03**: Interactive controls have consistent hover, focus-visible, active, disabled, loading/busy, and reduced-motion behavior.
- [x] **DS-04**: Forms and filters use consistent labels, help text, field errors, checkbox hit targets, input types, `autocomplete`, and `inputmode` where applicable.

### High-Leverage Admin Screens

- [x] **CLIENT-01**: Client detail becomes a scannable client workspace with clear identity, effective posture, credentials, endpoints, logout, DCR/RAT, and action sections.
- [x] **SUPPORT-01**: Token and consent index/detail screens support support-engineer incident work with scan-friendly rows, grouped metadata, safe action guidance, and cross-links to clients.
- [x] **OPERATE-01**: Interactions, device authorizations, and logout deliveries behave like operational queues with summaries, status/age/expiry context, empty states, and next-inspection links.
- [x] **SECURITY-01**: Policy pages surface security exceptions and override pressure, not only global counts.
- [x] **DCR-01**: Initial Access Token list/new/reveal screens feel like part of the DCR onboarding journey and show operator context for TTL, use, revocation, and partner handoff.
- [x] **KEYS-01**: Key generation and lifecycle actions explain impact, render visible errors, and use consistent confirmation/safety panels.

### Demo, Docs, Verification

- [x] **SEED-01**: Adoption demo seed data expresses every meaningful admin state needed for click-through and screenshots, including long names/URIs and incident states.
- [x] **DOCS-01**: Operator admin docs explain the final journey model and ownership boundary without turning into a protocol reference.
- [x] **VISUAL-01**: Browser-driven desktop and mobile screenshots cover the full admin surface, especially screens not covered by the previous pass.
- [x] **CONTRACT-01**: Tests fence the design system against regressions: no generic button classes, required admin CSS primitives exist, key journey links render, and raw inline-style clusters do not return on polished screens.

## Out of Scope

- Protocol, storage, or verifier behavior changes.
- A host theming engine or mandatory host branding system.
- Demo app visual redesign beyond seed data needed to exercise admin UI states.
- New public admin routes unless an existing workflow cannot be made clear in place.
- Tailwind migration or CSS architecture switch.
