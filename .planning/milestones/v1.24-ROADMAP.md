# Milestone v1.24: client_secret_jwt

**Status:** ✅ SHIPPED 2026-05-25
**Phases:** 88-90
**Total Plans:** 9

## Overview

This milestone closed the remaining practical direct-client authentication gap by adding a narrow `client_secret_jwt` slice on Lockspire-owned direct-client endpoints while preserving hashed-at-rest secret posture, strict replay and audience enforcement, and truthful operator and support surfaces.

## Phases

### Phase 88: Shared `client_secret_jwt` Runtime

**Goal**: Extend the shared direct-client authentication path so registered confidential clients can use `client_secret_jwt` consistently across the shipped Lockspire-owned direct-client endpoints.
**Depends on**: None
**Plans**: 3 plans

Plans:

- [x] 88-01: Add a narrow symmetric JWT verifier and explicit method routing in the shared direct-client auth pipeline
- [x] 88-02: Enforce issuer-string audience, bounded lifetime, replay, and algorithm posture for `client_secret_jwt`
- [x] 88-03: Prove valid and invalid `client_secret_jwt` behavior across representative Lockspire-owned direct-client surfaces

**Details:**
Phase 88 added the runtime slice itself. Shared direct-client auth now resolves JWT assertions from the registered client auth method after lookup instead of implicitly routing all JWT assertions through `private_key_jwt`, and endpoint-level allowlists keep `client_secret_jwt` on the intended Lockspire-owned direct-client surfaces only. The milestone also added sealed verifier material alongside the existing secret-hash posture so HS256 verification can succeed without widening raw-secret persistence, then proved success, replay, audience, algorithm, method-mismatch, and FAPI-denial behavior across representative shipped surfaces.

### Phase 89: Registration, Discovery, And Admin Truth

**Goal**: Make operator, DCR, discovery, and admin truth match the shipped `client_secret_jwt` runtime slice without widening package boundaries or weakening secret posture.
**Depends on**: Phase 88
**Plans**: 3 plans

Plans:

- [x] 89-01: Extend client registration and DCR validation to accept coherent `client_secret_jwt` metadata for confidential clients only
- [x] 89-02: Publish truthful discovery and endpoint auth-signing metadata for the new symmetric JWT slice
- [x] 89-03: Align admin/operator surfaces with the same stored auth-method and signing-alg truth while preserving redaction

**Details:**
Phase 89 made the new runtime support truthful everywhere Lockspire already describes or stores client auth method state. Client records now persist a typed `token_endpoint_auth_signing_alg` field, DCR and RFC 7592 flows require coherent `client_secret_jwt` plus `HS256` metadata for confidential clients only, discovery publishes the symmetric JWT slice only where the shared verifier actually runs, and admin/operator surfaces describe the same stored `client_secret_jwt` plus `HS256` truth without exposing verifier material or widening into a generic auth-metadata editor.

### Phase 90: Support Truth And Milestone Closure

**Goal**: Close the milestone with repo-native proof and documentation that describe the new symmetric JWT slice truthfully and narrowly.
**Depends on**: Phase 89
**Plans**: 3 plans

Plans:

- [x] 90-01: Update the canonical support-surface and host/operator docs for the shipped `client_secret_jwt` slice
- [x] 90-02: Add release-contract and support-truth proof that the docs and discovery metadata agree with runtime behavior
- [x] 90-03: Complete milestone-close verification and capture any explicitly deferred follow-on support work

**Details:**
Phase 90 aligned the public support contract, dedicated host guidance, onboarding docs, discovery metadata, runtime proof, and maintainer wording around one narrow `client_secret_jwt` story. Release-contract tests now pin the support semantics semantically instead of relying on prose snapshots, the docs explicitly keep `POST /par` excluded and FAPI/non-mTLS claims narrow, and the closeout artifacts capture both the full verification chain and the deferred follow-on support-burden work that remains outside this milestone.

## Milestone Summary

**Decimal Phases:**

- None

**Key Decisions:**

- Resolve direct-client JWT assertions from stored client auth-method truth after client lookup instead of treating every JWT assertion as `private_key_jwt`.
- Persist sealed verifier material for `client_secret_jwt` alongside hashed client secrets so HS256 verification can work without widening recoverable secret storage.
- Keep `client_secret_jwt` narrow: Lockspire-owned direct-client endpoints only, HS256 only, strict replay and issuer-string audience checks, and explicit FAPI denial.
- Store `token_endpoint_auth_signing_alg` as typed client state so DCR, RFC 7592, discovery, admin surfaces, and release docs all derive one coherent truth.

**Issues Resolved:**

- Confidential clients can now authenticate with `client_secret_jwt` on the shipped Lockspire-owned direct-client surfaces instead of being limited to password-style methods or asymmetric JWT auth.
- Invalid signature, replay, audience, expiry, method-mismatch, and disallowed-algorithm assertions now fail consistently as `invalid_client`.
- DCR, discovery, admin/operator surfaces, release-contract proof, and public docs now agree on the exact narrow support boundary for the symmetric JWT slice.

**Issues Deferred:**

- Broader symmetric JWT policy controls such as richer secret-rotation UX or wider algorithm-matrix support remain deferred unless real integrator demand emerges.
- Advanced setup support-burden reduction around `jwks_uri` rotation, mTLS, logout propagation, and protected-route setup remains the clearest next candidate after this milestone.

**Technical Debt Incurred:**

- None recorded at milestone close.
