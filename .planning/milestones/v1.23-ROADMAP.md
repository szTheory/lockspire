# Milestone v1.23: DCR Logout Metadata

**Status:** ✅ SHIPPED 2026-05-24
**Phases:** 85-87
**Total Plans:** 9

## Overview

This milestone removed the last practical self-service gap in Lockspire's existing logout propagation slice by letting DCR and RFC 7592 manage the four already-shipped logout metadata fields without widening the runtime support boundary.

## Phases

### Phase 85: DCR Intake And Representation

**Goal**: Extend DCR create and read flows so logout propagation metadata is accepted, persisted, and reflected from stored client state.
**Depends on**: None
**Plans**: 3 plans

Plans:

- [x] 85-01: Accept and validate logout propagation metadata during RFC 7591 registration
- [x] 85-02: Persist normalized logout metadata on typed client fields
- [x] 85-03: Expose stored logout metadata truthfully in DCR create and management-read responses

**Details:**
Phase 85 replaced the stale "unsupported" DCR posture with real intake, persistence, and readback support for `backchannel_logout_uri`, `backchannel_logout_session_required`, `frontchannel_logout_uri`, and `frontchannel_logout_session_required`. The work reused Lockspire's existing logout URI and origin validation rules, kept the values on typed `%Lockspire.Domain.Client{}` fields instead of extension metadata, and proved that both create and management-read responses serialize only the persisted logout truth.

### Phase 86: RFC 7592 Update Semantics And Proof

**Goal**: Add full-replace RFC 7592 update semantics for logout metadata while preserving existing registration lifecycle guarantees.
**Depends on**: Phase 85
**Plans**: 3 plans

Plans:

- [x] 86-01: Persist logout metadata on management `PUT` updates and clear omitted values under full-replace semantics
- [x] 86-02: Prove RAT rotation, persisted truth, provenance, and audit continuity for logout metadata updates
- [x] 86-03: Add positive and negative protocol/controller proof for logout metadata update behavior

**Details:**
Phase 86 brought the new logout metadata fields to the existing RFC 7592 write path without creating a second truth model. Management updates now reuse the same normalization path as create, omitted optional fields clear stored values under full-replace semantics, rotated registration access tokens remain truthful, and repo-native tests cover both success and invalid metadata flows across protocol and controller seams.

### Phase 87: Support Truth And Milestone Closure

**Goal**: Align the public support contract, operator guidance, DCR lifecycle docs, and milestone-close proof with the shipped logout metadata behavior.
**Depends on**: Phase 86
**Plans**: 3 plans

Plans:

- [x] 87-01: Correct the canonical support-surface statement for DCR-managed logout metadata
- [x] 87-02: Document create/read/update lifecycle behavior and RFC 7592 warnings in the DCR guide
- [x] 87-03: Align operator and maintainer release wording to the same canonical support contract

**Details:**
Phase 87 closed the truth gap around the new self-service slice. `docs/supported-surface.md` now states that DCR manages the four existing logout propagation metadata fields without claiming a broader logout runtime, `docs/dynamic-registration.md` explains the create/read/update lifecycle and full-replace semantics concretely, and operator plus maintainer docs now defer to one canonical support contract instead of drifting independently.

## Milestone Summary

**Decimal Phases:**

- None

**Key Decisions:**

- Reuse the same logout metadata validation and normalization primitives across operator flows, DCR create, and RFC 7592 updates.
- Persist logout propagation metadata on typed client fields rather than sidecar extension metadata.
- Drive create/read/update JSON truth directly from persisted client state instead of response-only reconstruction.
- Keep `docs/supported-surface.md` as the canonical public truth for what DCR/logout behavior Lockspire actually ships.

**Issues Resolved:**

- Self-service clients can now register, read, and replace the shipped logout propagation metadata without operator-only intervention.
- RFC 7592 full-replace semantics for logout metadata are now explicit and repo-proven, including clear-on-omit behavior and RAT rotation continuity.
- Public support, operator guidance, and maintainer release wording now agree on the same narrow logout support boundary.

**Issues Deferred:**

- `v1.24 client_secret_jwt` remains the next most practical direct-client auth gap.
- Additional operator/doctor coverage for advanced setup edges such as logout propagation, `jwks_uri` rotation, mTLS, and protected-route setup remains a follow-on support-cost candidate.

**Technical Debt Incurred:**

- None recorded at milestone close.
