# Milestone v1.6: Device Authorization Grant (RFC 8628)

**Status:** ✅ SHIPPED 2026-04-28
**Phases:** 30-32
**Total Plans:** 10

## Overview

Lockspire delivered the Device Authorization Grant wedge without widening beyond the embedded Phoenix library shape. The shipped surface now covers device-code issuance, a generated host-owned `/verify` seam, durable polling cadence enforcement, device-code redemption on `/token`, truthful discovery metadata, and generated-host end-to-end proof.

## Phases

### Phase 30: Core Device Authorization Endpoint & Storage

**Goal**: The provider can receive device authorization requests, generate codes, and store them securely with TTLs.
**Depends on**: Nothing
**Plans**: 3 plans

Plans:

- [x] 30-01: Create the device authorization domain model, secure Ecto schema, and repository storage.
- [x] 30-02: Add Base20/high-entropy code generation and the device authorization protocol pipeline.
- [x] 30-03: Expose the `/device/code` HTTP endpoint with RFC-shaped JSON and cache controls.

**Details:**
Phase 30 added the durable device-authorization domain, Ecto storage, Base20 `user_code` and high-entropy `device_code` generation, and the mounted `POST /device/code` HTTP surface. It also closed the remaining proof gap by adding a focused `mix test.phase30` alias and end-to-end coverage so the milestone no longer relied on manual UAT for issuance truth.

### Phase 31: Host-Owned Verification UI Seam

**Goal**: The host application has the integration seams and documentation needed to build a secure user verification UI.
**Depends on**: Phase 30
**Plans**: 4 plans

Plans:

- [x] 31-01: Extend device-authorization lifecycle state and race-safe repository transitions for verification.
- [x] 31-02: Generate the host-owned `/verify` router and controller starter seam with explicit anti-phishing behavior.
- [x] 31-03: Publish the device-flow host guide and wire onboarding and supported-surface docs to the Phase 31 seam.
- [x] 31-04: Add the narrow verification protocol API, prefill response field, and executable response-surface proof.

**Details:**
Phase 31 introduced durable verification handles, typed pending and terminal verification outcomes, the generated host-owned `/verify` controller/template seam, and the documentation contract that keeps `verification_uri_complete` prefill-only. The shipped generator output now includes CSRF-protected forms and repo-backed runtime fixtures that prove the generated seam matches the expected host-facing contract.

### Phase 32: Polling & Token Issuance

**Goal**: Devices can poll the token endpoint and receive tokens once the user authorizes the request.
**Depends on**: Phase 31
**Plans**: 3 plans

Plans:

- [x] 32-01: Extend device-authorization storage with durable poll-window state, sticky `slow_down`, and single-winner consume semantics.
- [x] 32-02: Add the device grant branch to `TokenExchange` and reuse shared token issuance for RFC-shaped polling outcomes.
- [x] 32-03: Prove the HTTP, discovery, and docs contract and add end-to-end device-flow token redemption coverage.

**Details:**
Phase 32 made polling state durable across nodes, mapped RFC 8628 continuation and terminal outcomes onto the shared `/token` pipeline, and published truthful discovery and docs support for the device-flow slice. The final proof surface now exercises `/device/code -> /verify -> /token` end to end, including replay collapse and generated-host verification URI wiring.

## Milestone Summary

**Key Decisions:**

- Keep the Device Authorization Grant as an embedded-library wedge that reuses Lockspire's existing token issuance pipeline instead of adding a separate hosted auth surface.
- Preserve the host-owned verification seam: `/verify` is generated and editable host code, not a Lockspire-owned browser UI.
- Keep polling truth durable in Postgres via row-locked repository operations rather than process-local pacing state.
- Advertise the device grant and device authorization endpoint only because the mounted routes, docs, and generated-host proof now agree.

**Issues Resolved:**

- Closed the missing Phase 30 automation gap with maintained end-to-end issuance proof.
- Reconciled verification lookup so repository-backed `/verify` flows can re-display a real normalized device code.
- Added CSRF protection to the generated verification forms without weakening the explicit review-step contract.
- Cleared milestone-close audit and Nyquist drift so the device-flow release claim is backed by current repo truth.

**Issues Deferred:**

- No new device-flow runtime gaps were deferred at milestone close.

**Technical Debt Incurred:**

- Public product posture remains preview even though the shipped device-flow surface is now fully archived and verified.
