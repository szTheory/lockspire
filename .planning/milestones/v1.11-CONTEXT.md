# v1.11 Private Key JWT Client Authentication

## Project Description
Lockspire is an embedded OAuth/OIDC provider library for Elixir/Phoenix applications. This milestone implements Private Key JWT Client Authentication (RFC 7523), allowing clients to authenticate using asymmetric signatures instead of shared secrets.

## Why This Milestone
In the v1.10 milestone, Lockspire shipped strict-mode enforcement for the FAPI 2.0 Security Profile. However, FAPI 2.0 strictly requires either mTLS or Private Key JWT for client authentication, as shared secrets are forbidden. Because mTLS is permanently out of scope for Lockspire (due to proxy termination complexities in standard Phoenix deployments), Lockspire's current FAPI 2.0 claim remains incomplete in practice. Implementing Private Key JWT is the mandatory, non-negotiable next step to close this gap and allow real FAPI 2.0 clients to authenticate securely, making the provider truly production-ready for high-security environments.

## User-Visible Outcome
Operators and dynamic clients can register with `token_endpoint_auth_method: "private_key_jwt"` and configure their public keys via `jwks` or `jwks_uri`. During token or PAR requests, the client authenticates by sending a `client_assertion` JWT signed with its private key. The Lockspire provider verifies the signature against the client's public key, successfully eliminating the need for shared secrets.

## Completion Class
Integration and Protocol Conformance. Requires database migrations, HTTP client integrations, and strict protocol compliance.

## Final Integrated Acceptance
A FAPI 2.0 client successfully pushes an authorization request and exchanges a code for tokens using a `client_assertion` signed with its private key. Lockspire successfully fetches and caches the client's public key from a remote `jwks_uri`, validates the signature, and correctly rejects any immediate replay of the same `jti` in a subsequent request.

## Architectural Decisions

### `jti` Replay Tracking via Ecto
- **Rationale:** In a BEAM cluster, in-memory ETS caches for JWT IDs (`jti`) fail to protect against replays across nodes. We will use a dedicated Ecto-backed database table (`lockspire_used_jtis`), pruned asynchronously via Oban, to provide stateless, cluster-safe replay protection.
- **Alternatives:** Mnesia/ETS (rejected due to cluster distribution complexity in typical Phoenix deployments), Redis (rejected to minimize infrastructure dependencies).

### Aggressive `jwks_uri` Caching via Finch
- **Rationale:** Dynamic key fetching during the authentication path introduces a network tar pit and potential denial-of-service vector. Fetching must use Finch with strict timeouts and aggressive local caching.
- **Alternatives:** Fetching on every request (rejected due to latency and DoS risk), requiring static `jwks` only (rejected as it limits interoperability and key rotation).

### Permissive Audience Validation
- **Rationale:** Overly strict `aud` validation is the most common footgun in Node/Keycloak ecosystems. To optimize developer ergonomics, Lockspire will accept both the exact Token Endpoint URL and the Issuer URL as valid audiences in the client assertion.
- **Alternatives:** Strict Token Endpoint URL matching only (rejected due to poor DX).

## Error Handling Strategy
- Explicit rejection of symmetric algorithms (`HS256`, `HS384`, etc.) and `none` to prevent algorithm confusion attacks.
- Strict timeout and circuit-breaker handling for HTTP key fetches.
- Clear `invalid_client` OAuth error responses with descriptive (but not overly leaky) messages when a `jti` is replayed or a signature is invalid.

## Risks and Unknowns
- Dynamic JWKS fetching introduces a synchronous external network dependency during the critical authentication path, posing a latency risk if the cache is cold and the upstream server is slow.
- The Oban pruner for the `jti` table must be carefully tuned to prevent database lock contention on high-volume providers.

## Existing Codebase / Prior Art
- `Lockspire.Protocol.ClientAuth`: Current home for symmetric and `none` client authentication.
- `Lockspire.Protocol.Registration`: Handles DCR, needs updating to accept `private_key_jwt` and `jwks`/`jwks_uri` parameters.
- `JOSE` and `Joken`: Existing dependencies used for JWT parsing and verification.
- `Lockspire.Storage.Ecto.ClientRecord`: Needs schema updates for the new auth method.

## Relevant Requirements
- FAPI 2.0 Client Authentication
- RFC 7523 (JSON Web Token (JWT) Profile for OAuth 2.0 Client Authentication)

## Scope
**In Scope:**
- Validation of `private_key_jwt` client assertions.
- Support for `jwks` (by-value) and `jwks_uri` (by-reference) client metadata.
- Dynamic JWKS fetching, parsing, and caching.
- Ecto-backed `jti` replay protection table and Oban pruner.
- Updates to DCR and Admin UI for the new auth method.

**Out of Scope:**
- mTLS client authentication.
- Outbound signed requests (Lockspire acting as a client to another service).

## Technical Constraints
- Must remain cluster-safe; no local-node-only state for critical security checks like replays.
- Must not introduce new heavy external dependencies beyond what a standard Phoenix/Ecto app already runs.

## Integration Points
- External client `jwks_uri` endpoints (HTTP GET).
- Existing Oban supervision tree for background pruning.

## Testing Requirements
- Unit tests for `client_assertion` parsing, validation, and algorithm restriction.
- Integration tests for `jwks_uri` fetching, caching behavior, and timeout fallbacks.
- End-to-End test of the full authorization code flow where the client uses `private_key_jwt` at the PAR and Token endpoints.
- Security tests verifying that reused `jti` values are rejected and `HS256` signatures are rejected.

## Acceptance Criteria
1. `private_key_jwt` is an accepted and persisted `token_endpoint_auth_method` in DCR and the Admin UI.
2. Clients can configure `jwks_uri` or `jwks` in their registration.
3. Token and PAR requests with a valid `client_assertion` JWT and `client_assertion_type=urn:ietf:params:oauth:client-assertion-type:jwt-bearer` succeed.
4. Token and PAR requests reusing a `jti` within the token's lifetime are strictly rejected.
5. Token and PAR requests signed with `HS256` or `none` are strictly rejected.
6. Dynamic JWKS fetching uses Finch with a strict timeout and caches responses locally.

## Open Questions
- Should we provide a mix task or admin UI tool to help developers debug invalid `client_assertion` claims locally, given how opaque JWT signature failures can be?
