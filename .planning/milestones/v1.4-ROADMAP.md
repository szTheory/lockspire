# Milestone v1.4: JAR and Request Objects

**Status:** ✅ SHIPPED 2026-04-26
**Phases:** 21-24
**Total Plans:** 18

## Overview

Lockspire completed its fourth planning milestone by adding JWT Secured Authorization Requests (JAR) support on top of the existing embedded OAuth/OIDC provider surface. The milestone stayed narrow: signed request objects, request-object validation, discovery truth, operator policy controls, and final closure proof, while keeping JAR-04 decryption explicitly deferred.

This archive records the milestone as complete from a planning and implementation standpoint. The package version in `mix.exs` remains `0.2.0` preview posture at archive time, but the shipped surface now includes the verified JAR slice alongside the earlier PAR and PAR-policy work.

## Milestone Summary

**Key Accomplishments:**

- Added signed JAR request-object support with validation against client keys and RFC 9101 claim checks.
- Integrated request objects into the existing `/authorize` and PAR flow without widening the embedded-library shape.
- Published truthful JAR discovery and operator policy controls for required-JAR enforcement.
- Closed the milestone with explicit requirement traceability and validation proof.
- Preserved JAR-04 decryption as a deferred future requirement.

**Key Decisions:**

- Keep v1.4 narrow around JAR-by-value on the existing authorization-code and PAR path rather than broadening into a larger request-object or grant-family expansion.
- Reuse the existing client-key infrastructure for request-object signature validation.
- Preserve the host-app seam and avoid introducing hosted-auth or end-user authentication responsibilities into Lockspire.
- Keep JAR-04 explicitly deferred so the shipped milestone stays truthful about supported request-object behavior.

**Issues Resolved:**

- Closed the shipped JAR requirement set with traceability instead of treating the validation surface as implied.
- Kept discovery and docs aligned with the actual supported JAR slice.
- Reconciled milestone state so closure reads cleanly across requirements, roadmap, and project state.

**Issues Deferred:**

- JAR decryption remains out of scope and deferred for future planning.
- The next milestone still needs to decide which protocol or operator-surface expansion should follow JAR.

**Technical Debt Incurred:**

- No new blocking technical debt was accepted in the milestone close.
