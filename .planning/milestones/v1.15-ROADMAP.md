# Milestone v1.15: JWKS URI & Private Key JWT Client Authentication

**Status:** ✅ SHIPPED 2026-05-06
**Phases:** 59-62
**Total Plans:** 13

## Overview

v1.15 closed the strongest remaining confidential-client authentication gap in Lockspire’s embedded-provider surface. The milestone kept scope narrow and truthful: it admits guarded `jwks_uri` registration only within the supported `private_key_jwt` slice, hardens remote JWKS resolution into an explicit security boundary, upgrades `private_key_jwt` from structural acceptance to shared cryptographic verification, and proves the resulting runtime truth through discovery, docs, telemetry, and end-to-end release checks.

## Phases

### Phase 59: Registration, Policy & Metadata Truth
**Goal**: Clients and operators can configure the `private_key_jwt` slice truthfully, and discovery/endpoint metadata advertise only what Lockspire will actually verify.
**Depends on**: None (v1.14 completed)
**Requirements**: REG-01, REG-02, REG-03, META-01, META-02
**Plans**: 3 plans

Plans:

- [x] 59-01: DCR and RFC 7592 `jwks_uri` intake/update rules for the narrow client-auth slice
- [x] 59-02: Admin/policy truth for `private_key_jwt` and assertion signing algorithms
- [x] 59-03: Discovery, revocation, and introspection auth-method metadata alignment

**Details:**
Phase 59 defined the milestone contract. It kept the remote-key story bounded to Lockspire’s direct client-auth wedge and aligned DCR, admin policy, and metadata truth before the fetcher or verifier work began.

### Phase 60: Guarded Remote JWKS Resolution
**Goal**: Lockspire can retrieve client key material from `jwks_uri` safely enough to stay embedded and trustworthy.
**Depends on**: Phase 59
**Requirements**: JWKS-01, JWKS-02, JWKS-03
**Plans**: 3 plans

Plans:

- [x] 60-01: Harden `Lockspire.JwksFetcher` request policy and failure model
- [x] 60-02: Enforce network-target safety and response-size boundaries
- [x] 60-03: Cache TTL, refresh, and key-rotation recovery behavior

**Details:**
Phase 60 turned the existing fetcher into a narrow security boundary rather than a convenience helper. Remote JWKS lookup is now `https`-only, redirect-free, retry-free, target-safe, payload-bounded, and rotation-aware without widening into a generic outbound fetch subsystem.

### Phase 61: Shared Private Key JWT Verification
**Goal**: All Lockspire-owned direct client-auth surfaces enforce full `private_key_jwt` verification consistently.
**Depends on**: Phase 60
**Requirements**: PKJWT-01, PKJWT-02, PKJWT-03, PKJWT-04, PKJWT-05, PKJWT-06, OBS-01
**Plans**: 4 plans

Plans:

- [x] 61-01: Split `ClientAuth` into lookup, key resolution, signature verification, and claims validation stages
- [x] 61-02: Enforce issuer-bound audience, algorithm allowlists, TTL/skew, and replay ordering
- [x] 61-03: Wire verified `private_key_jwt` behavior across token-adjacent direct-client endpoints
- [x] 61-04: Telemetry, audit, and redaction proof for auth failures and replay outcomes

**Details:**
Phase 61 is the core milestone delivery. It converted Lockspire’s previous partial implementation into a truthful one by making signature trust, claim validation, replay ordering, direct-client surface wiring, and failure observability explicit and shared rather than endpoint-specific.

### Phase 62: Docs, Verification & Closure
**Goal**: The shipped client-auth surface is understandable, executable, and release-truthful.
**Depends on**: Phase 61
**Requirements**: DOC-01, V-01, V-02, V-03

Plans:

- [x] 62-02: End-to-end and negative-path verification across representative direct-client endpoints
- [x] 62-03: Release-contract, traceability, and milestone-closure proof

**Details:**
Phase 62 closed the milestone the same way Lockspire closes other trust-sensitive work: executable proof first, then support-surface truth, then release-contract and planning closure.

## Milestone Summary

**Decimal Phases:**

- None

**Key Decisions:**

- Kept the milestone limited to `jwks_uri` plus `private_key_jwt`; did not widen into `client_secret_jwt`, mTLS, or federation trust chains.
- Used issuer-identifier audience validation for `private_key_jwt` rather than permissive endpoint-based audience matching.
- Reused Lockspire’s shared `ClientAuth` seam so one hardened implementation raises trust across all direct-client endpoint surfaces that already depend on it.

**Issues Resolved:**

- Replaced payload-shape-only `private_key_jwt` acceptance with shared cryptographic verification and trusted-claim enforcement.
- Added guarded remote JWKS resolution with SSRF-resistant target checks, bounded payload handling, and explicit cache-refresh semantics.
- Aligned discovery, docs, telemetry, audit, and release-contract proof to the shipped authentication surface.

**Issues Deferred:**

- `client_secret_jwt`
- mTLS client authentication
- signed metadata / federation-style key trust
- Historical `37-VERIFICATION.md` debt remains tracked outside this milestone in `STATE.md`

**Technical Debt Incurred:**

- Phase 61 validation bookkeeping does not yet use the newer Nyquist frontmatter/sign-off shape.
- Phase 62 relies on executable verification and release-contract proof without a separate `62-VALIDATION.md` artifact.
