# Roadmap: v1.20 Mutual TLS (RFC 8705)

## Overview

This milestone implements Mutual TLS (mTLS) Client Authentication and Certificate-Bound Access Tokens (RFC 8705). Given Lockspire's embedded nature behind reverse proxies, this implementation focuses on extracting proxy-terminated client certificates via host seams. It provides advanced sender-constrained tokens and authentication mechanisms required in high-security ecosystems (like OpenBanking/FAPI), matching the capabilities of DPoP but leveraging transport-layer cryptography properties passed through infrastructure boundaries.

## Phases

- [ ] **Phase 75: mTLS Infrastructure & Header Seams** - Safely extract and parse proxy-forwarded client certificates
- [ ] **Phase 76: `self_signed_tls_client_auth`** - Enable client authentication using self-signed mTLS certificates
- [ ] **Phase 77: `tls_client_auth` PKI Authentication** - Enable client authentication matched against registered Subject DNs or SANs
- [ ] **Phase 78: Certificate-Bound Access Tokens** - Issue and validate tokens cryptographically bound to the client's TLS session
- [ ] **Phase 79: Introspection & Discovery Surface** - Advertise mTLS capabilities and expose bindings for resource servers

## Phase Details

### Phase 75: mTLS Infrastructure & Header Seams
**Goal**: The Elixir runtime can safely extract, parse, and thumbprint proxy-forwarded client certificates via a configurable host seam.
**Depends on**: None
**Requirements**: MTLS-01, MTLS-02
**Success Criteria** (what must be TRUE):
  1. Host apps can configure a custom header or Plug to supply the client certificate.
  2. The library correctly parses PEM certificates and calculates the `x5t#S256` thumbprint.
  3. Requests without valid certificates fail safely or are treated as anonymous.
**Plans**: TBD

### Phase 76: `self_signed_tls_client_auth`
**Goal**: Clients can authenticate to the token endpoint using self-signed mTLS certificates.
**Depends on**: Phase 75
**Requirements**: MTLS-03
**Success Criteria** (what must be TRUE):
  1. Clients can register with `token_endpoint_auth_method: "self_signed_tls_client_auth"`.
  2. Token requests using this method validate the presented mTLS certificate against the client's registered JWKS.
  3. Authentication fails if the certificate does not match the registered keys.
**Plans**: TBD

### Phase 77: `tls_client_auth` PKI Authentication
**Goal**: Clients can authenticate using PKI-issued mTLS certificates matched against registered Subject DNs or SANs.
**Depends on**: Phase 75
**Requirements**: MTLS-04
**Success Criteria** (what must be TRUE):
  1. Clients can register with `token_endpoint_auth_method: "tls_client_auth"` and relevant metadata attributes.
  2. Token requests validate the presented certificate's subject/SAN against the registered metadata.
  3. Authentication fails if the certificate attributes do not match the expected values.
**Plans**: TBD

### Phase 78: Certificate-Bound Access Tokens
**Goal**: Access and refresh tokens are securely bound to the client's TLS certificate.
**Depends on**: Phase 76, Phase 77
**Requirements**: MTLS-05, MTLS-06
**Success Criteria** (what must be TRUE):
  1. Tokens issued via mTLS client authentication include the `cnf` claim with the `x5t#S256` thumbprint.
  2. Refresh token requests reject the grant if the presented certificate's thumbprint does not match the binding.
  3. DPoP and mTLS bindings do not conflict (if both are present).
**Plans**: TBD

### Phase 79: Introspection & Discovery Surface
**Goal**: The authorization server correctly advertises mTLS capabilities and exposes bindings for resource servers.
**Depends on**: Phase 78
**Requirements**: MTLS-07, MTLS-08
**Success Criteria** (what must be TRUE):
  1. `/.well-known/openid-configuration` advertises mTLS support, supported auth methods, and `mtls_endpoint_aliases`.
  2. The introspection endpoint response includes the `cnf` claim with `x5t#S256` when inspecting an mTLS-bound token.
  3. End-to-end mTLS conformance tests pass for both client authentication and token binding.
