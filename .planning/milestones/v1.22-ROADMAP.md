# Milestone v1.22: DPoP Nonce Support

**Status:** ✅ SHIPPED 2026-05-24
**Phases:** 82-84
**Total Plans:** 8

## Overview

This milestone closed the last major DPoP trust gap in Lockspire's already-shipped sender-constrained story by adding server-issued nonce challenge and retry behavior to the exact surfaces Lockspire already owns and proves: `/token`, `/userinfo`, and the canonical host Phoenix protected-route pipeline.

## Phases

### Phase 82: Shared DPoP Nonce Primitive

**Goal**: Add one shared DPoP nonce issuance and validation path without introducing new operator or client policy knobs.
**Depends on**: None
**Plans**: 2 plans

Plans:

- [x] 82-01: Add a shared DPoP nonce primitive and wire nonce validation into the existing proof validator
- [x] 82-02: Add unit proof for nonce issuance, purpose separation, and typed nonce failure reasons

**Details:**
Phase 82 established one protocol-owned nonce primitive for both authorization-server and resource-server DPoP validation. The work kept nonce state out of storage and config by issuing signed opaque values from `secret_key_base`, preserved strict purpose separation between token and protected-resource surfaces, and surfaced stable `:missing_dpop_nonce` / `:invalid_dpop_nonce` failures through the shared validator so later HTTP adapters could stay thin.

### Phase 83: Lockspire-owned DPoP Endpoint Adoption

**Goal**: Apply nonce challenge/retry behavior to the Lockspire-owned DPoP token and protected-resource surfaces.
**Depends on**: Phase 82
**Plans**: 3 plans

Plans:

- [x] 83-01: Add authorization-server nonce challenge and retry behavior on `/token`
- [x] 83-02: Add resource-server nonce challenge and retry behavior on `/userinfo`
- [x] 83-03: Keep replay, `ath`, binding, MTLS, and bearer regressions covered while adopting nonce support

**Details:**
Phase 83 adopted the shared nonce seam on Lockspire-owned public surfaces. `/token` now returns retryable `use_dpop_nonce` failures and accepts retried authorization-code, device-code, CIBA, and refresh exchanges with the issued authorization-server nonce. `/userinfo` now returns the exact DPoP `401` challenge contract with `DPoP-Nonce`, and the regression suite kept replay, binding, `ath`, MTLS, and bearer semantics pinned while nonce support landed.

### Phase 84: Host Plug Pipeline, Docs, and Milestone Closure

**Goal**: Extend the shipped host Phoenix plug contract and public support story to include nonce-backed DPoP.
**Depends on**: Phase 83
**Plans**: 3 plans

Plans:

- [x] 84-01: Add nonce-aware DPoP challenge behavior to `EnforceSenderConstraints` and `RequireToken`
- [x] 84-02: Update supported-surface and protected-route docs plus release-readiness contract wording
- [x] 84-03: Prove the generated-host protected-route nonce retry path and close the milestone

**Details:**
Phase 84 completed the shipped host-route story. `EnforceSenderConstraints` now passes `conn.secret_key_base` into protected-resource DPoP validation, `/userinfo` and host plugs share one protected-resource challenge helper, the public docs now claim only the shipped nonce-backed DPoP surface, and the generated-host Phoenix route proves the real retry path end to end.

## Milestone Summary

**Decimal Phases:**

- None

**Key Decisions:**

- Keep DPoP nonce issuance stateless and rooted in `secret_key_base` instead of adding storage or policy knobs.
- Reuse the shared DPoP validator and existing token/protected-resource seams rather than introducing endpoint-specific nonce validators.
- Share protected-resource challenge transport between `/userinfo` and host Phoenix plugs, while keeping protocol validation ownership inside `Lockspire.Protocol.ProtectedResourceDPoP`.
- Narrow public support claims to Lockspire-owned `/token`, Lockspire-owned protected resources, and host Phoenix routes protected by the shipped plug pipeline.

**Issues Resolved:**

- Closed the last major DPoP sender-constrained trust gap on already-shipped Lockspire surfaces.
- Added truthful `use_dpop_nonce` retry behavior to both token issuance and protected-resource validation paths.
- Proved the canonical host Phoenix protected-route retry path instead of leaving nonce support as a plug-only claim.

**Issues Deferred:**

- DCR logout propagation metadata remains the next highest-leverage candidate for self-service client ergonomics.

**Technical Debt Incurred:**

- None recorded at milestone close.
